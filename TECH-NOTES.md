## 1. Механизм кодов подтверждения в режиме «разработка»

Код подтверждения — защита от опасных действий агента (правки файлов, запуск скриптов) в режиме `dev`.

### 1.1 Формирование плана изменений

- Ядро агента: `agent/core.ts` (`runAgentCore()`).
- При работе в режиме `dev` агент анализирует tool calls от LLM:
  - если среди них есть `write_file` или `run_command`, они не выполняются сразу;
  - вместо этого формируется список действий (`actions`) с человеко‑читаемыми описаниями:
    - для `write_file` — путь файла + описание из `getWriteFileDescription()`;
    - для `run_command` — команда + пояснение из `getCommandDescription()` (например, «Сборка приложения», «Применение миграций БД», «Резервная копия файлов»).
- В `agent/core.ts`:
  - из массива tool calls выделяются только модифицирующие;
  - дополнительно собирается список `filesToBackup` (пути файлов/каталогов, которые нужно сохранить перед выполнением команд, особенно при `prisma migrate`).

### 1.2 Хранилище ожиданий (approval-store)

- Файл: `agent/approval-store.ts`.
- Типы:
  - `PendingApproval` — ожидает первичного подтверждения плана:
    - `code` — 4‑значный код;
    - `actions` — текстовое описание шагов;
    - `toolCalls` — сериализованные tool calls (имя, аргументы);
    - `messages` — упрощённая история сообщений для продолжения диалога после выполнения;
    - `filesToBackup` — файлы/каталоги для бэкапа;
    - `createdAt`, `lastShownAt` — таймстемпы для TTL.
  - `PendingVerification` — ожидает окончательного принятия/отката после выполнения плана:
    - `backupId` — идентификатор резервной копии от `scripts/agent-backup.ts`;
    - `executionResult` — сводка результатов работы инструментов;
    - `createdAt`, `lastShownAt`.
- Хранилище:
  - В памяти: `Map<string, Pending>`.
  - На диске: файл `.agent/pending-codes.json` в корне репозитория:
    - путь вычисляется относительно `getConfig().root` (`AGENT_ROOT` или корневой каталог проекта);
    - при старте агента выполняется `loadStore()`, при каждом изменении — `saveStore()`.
  - TTL: `APPROVAL_TTL_MS = 30 * 60 * 1000` (30 минут).
    - Момент отсчёта TTL: `ttlStart(p)` — `lastShownAt ?? createdAt`.
    - При загрузке хранилища устаревшие записи отбрасываются.
- Безопасная сериализация:
  - `pendingToSerializable()` и `pendingFromSerializable()` хранят в JSON только сериализуемые данные (role/content/tool_calls), чтобы исключить падения `JSON.stringify` из‑за циклических ссылок или сложных объектов.
  - При ошибках чтения/записи в лог пишутся сообщения `[approval-store] loadStore failed` / `saveStore failed`.

### 1.3 Жизненный цикл кода

Управление кодами реализовано в `agent/core.ts` и `agent/approval-store.ts`.

**1. Показ плана и генерация кода**

1. Агент получает промпт в режиме `dev`.
2. LLM возвращает текст+tool calls с модифицирующими действиями.
3. Агент:
   - формирует список `actions` (читайте‑план для человека);
   - собирает `toolCalls` и `filesToBackup`;
   - генерирует 4‑значный код через `generateCode()` (`1000–9999`);
   - сохраняет `PendingApproval` в память и на диск (`setPending()`), при этом `lastShownAt` устанавливается в текущий момент.
4. Пользователь получает сообщение с планом:
   - подробный список действий;
   - текст «Подтвердите кодом: XXXX» и инструкцию «отправьте 4 цифры».

**2. Подтверждение плана (первый ввод кода)**

1. В режиме `dev` каждое входящее сообщение рассматривается как возможный код:
   - принимаются варианты:
     - точно `XXXX`;
     - `код 1234`, `Код: 1234`, `подтверждаю 1234` и подобные;
     - распознавание реализовано регулярными выражениями в `runAgentCore`.
2. Функция `consumePending(code)`:
   - загружает хранилище (при необходимости) и очищает истёкшие записи;
   - достаёт запись по коду, помечает, истёк ли TTL;
   - удаляет запись из памяти/диска и возвращает `{ pending, expired }`.
3. Ветвления:
   - если `pending.kind === "approval"` и `expired === false`:
     - агента выполняет резервное копирование (`npx tsx scripts/agent-backup.ts backup <filesToBackup>`), сохраняет `backupId`;
     - по очереди выполняет все tool calls через `executeTool()` (`write_file`, `run_command`);
     - собирает краткий отчёт по каждому шагу;
     - генерирует новый код `verifyCode` («код верификации»), сохраняет `PendingVerification` с `backupId` и `executionResult`;
     - отвечает пользователю текстом:
       - «Действия выполнены. Проверьте результат…»;
       - код для окончательного принятия и фраза для отката (`откат XXXX`).
   - если `pending.kind === "approval"` и `expired === true`:
     - возвращает «Время подтверждения истекло. Повторите запрос для нового кода.».
   - если `pending.kind === "verification"`:
     - при вводе кода без `откат` считается, что изменения принимаются окончательно;
     - пользователю возвращается «Изменения приняты» + `executionResult`.

**3. Откат (`откат XXXX`)**

- В `runAgentCore` есть отдельная ветка для строк вида `откат 1234`:
  - по коду вызывается `consumePending()`;
  - если `pending.kind === "verification"` и есть `backupId`, выполняется:
    - `npx tsx scripts/agent-backup.ts restore <backupId>`;
    - при успехе возвращается «Откат выполнен…»;
    - при отсутствии резервной копии — «Откат невозможен: нет резервной копии.».
- Если код не относится к `verification`, возвращается «Код не для отката».

**4. TTL от последнего показа кода**

- Каждый раз, когда модель в тексте ответа снова показывает код (например при уточнении плана), `runAgentCore` выдёргивает его из строки результата и вызывает `refreshPendingShown(code)`.
- TTL 30 минут считается от `lastShownAt` (момент последнего показа), а не от первоначального `createdAt`.  
Это позволяет вести диалог вокруг одного плана, не сбрасывая код преждевременно.

### 1.4 Исправленные проблемы по кодам (сводка из истории)

См. подробности в `what's new.md` (записи от 04.03.2026 и 27–28.02.2026). Кратко:

- **Коды терялись при рестарте агента**:
  - Причина: хранилище было только in‑memory.
  - Решение: персистентный `approval-store` с файлом `.agent/pending-codes.json`, загрузкой при старте и TTL.
- **TTL считался от момента создания, а не от последней демонстрации плана**:
  - Причина: при длинных диалогах или уточнениях пользователь не успевал подтвердить код.
  - Решение: введено поле `lastShownAt` и функция `refreshPendingShown()`, TTL считается от последнего показа.
- **Код не принимался сразу после показа плана**:
  - Причины:
    - при сохранении в JSON ломалась сериализация сложного объекта `messages`, файл не записывался;
    - промпт типа `код 2277` или `подтверждаю 2277` не распознавался как код.
  - Решение:
    - введена безопасная сериализация (`pendingToSerializable`/`pendingFromSerializable`);
    - логирование ошибок `loadStore`/`saveStore`;
    - расширено распознавание форматов ввода в `agent/core.ts` и `agent/contract.ts`;
    - кодовые промпты обходят кэш агента (см. раздел 2.3).
- **Ошибка «Неверный код подтверждения» из‑за кэша**:
  - Причина: кэш `agent_prompt_cache` возвращал старый ответ «Неверный код» вместо реальной проверки в `approval-store`.
  - Решение: в `agent/contract.ts` промпты вида `XXXX` и `откат XXXX` больше не используют кэш (forceFresh).
- **Агент падал при выполнении подтверждённого плана**:
  - Причина: некорректная структура `toolCalls`/`filesToBackup` после восстановления из JSON или ошибки в конкретном tool.
  - Решение:
    - обёртка выполнения плана в `try/catch`;
    - проверки на непустые массивы и корректный тип `tc.arguments`;
    - при ошибке агент возвращает текст с деталями вместо завершения процесса.

## 2. Режимы агента и системный промпт

### 2.1 Режимы: chat / consult / dev

- Определены в типе `AgentMode` (`agent/llm/system-prompt.ts`) и используются в `agent/core.ts`.
- Поведение:
  - **chat (`курилка`)**:
    - системный промпт упрощён, без «жёстких правил» и без доступа к коду/файлам;
    - в `agent/core.ts` системный промпт вообще не отправляется в модель (экономия токенов и «чистый чат»);
    - в `agent/tools/index.ts` в этом режиме разрешены только `get_agent_info` и `write_docs_file` (создание/обновление текстовых файлов в `docs/`), любые обращения к коду блокируются с ошибкой.
  - **consult**:
    - доступно чтение кода, поиск по файлам, запросы к API;
    - инструменты, модифицирующие код (`write_file`, `run_command`), используются только если явно разрешено в промпте, но без режима `dev` агент их не применяет.
  - **dev**:
    - полный доступ к инструментам: чтение/запись файлов, `run_command` из белого списка, Prisma‑команды;
    - модифицирующие действия всегда проходят через механизм кодов подтверждения (см. раздел 1).

### 2.2 Отправка системного промпта

Логика в `agent/core.ts`:

- В режимах `consult` и `dev`:
  - при **первом** сообщении в чате (история пуста) системный промпт отправляется как первое сообщение `role: "system"`;
  - при последующих сообщениях в том же чате промпт **не дублируется** — модель опирается на память и историю переписки;
  - в логе (`logEntries`) фиксируется, отправлялся ли системный промпт.
- В режиме `chat`:
  - системный промпт для модели вообще не используется (агент пишет об этом в лог: «РЕЖИМ КУРИЛКА: системный промпт НЕ используется»).

Цели:
- снизить расход токенов на повторяющийся системный контекст;
- сохранить акцент на переписке пользователя и фактическом коде проекта;
- не перегружать модель регламентом в лёгком режиме «курилка».

### 2.3 Кэш запросов и обход для кодов

- Кэш: `agent/cache/*`, таблица `agent_prompt_cache` в `prisma/schema.prisma`.
  - Сохраняет пары `prompt`/`response` вместе с метаданными (модель, режим, проект, пользователь, длины).
  - Поиск похожих промптов по текстовому сходству; порог читабельности задаётся `cacheSimilarityThreshold` (в конфиге агента).
- Для промптов вида:
  - `^\d{4}$` (чистый 4‑значный код);
  - `^откат\s+\d{4}$` и их варианты;
агент **всегда** делает свежий запрос в `runAgentCore`, минуя кэш. Это предотвращает возвращение устаревших ответов («Неверный код» от старой сессии).

## 3. Seed‑скрипты и тестовые данные

### 3.1 Общий демо‑сид (seed-demo)

- Отчёт: `scripts/SEED-DEMO-REPORT.md`.
- Основной скрипт: `scripts/seed-demo.ts` (описан в отчёте, код не дублируется здесь).
- Действия:
  - очищает `items`, `news`, `testimonials` (каскадно удаляя связанные `cart_items` и `messages`);
  - создаёт:
    - администратора `bshevelev@mail.ru` с паролем `123456`;
    - несколько демо‑пользователей;
    - набор новостей и отзывов;
    - десяток товаров с описанием и плейсхолдер‑картинками.
- Может запускаться многократно, переиспользуя существующих пользователей по email.

### 3.2 `scripts/seed-five-users.ts`

- Цель: быстро создать «живой» каталог с реальными профилями и товарами.
- Поведение:
  - создаёт 5 пользователей с паролем `123456` (если их ещё нет);
  - для каждого — профиль `profiles` с аватаркой `https://placehold.co/150?...`;
  - каждому пользователю создаются 3 товара с тремя картинками (placehold.co по названию товара).
- Особенности:
  - используется `bcryptjs` для хеширования паролей;
  - при повторном запуске не дублирует пользователей — существующие email‑ы пропускаются;
  - ссылки на изображения ведут на внешний сервис (`placehold.co`), но для продакшена предпочтительно иметь локальные копии (см. следующий пункт).

### 3.3 `scripts/recreate-items.ts`

- Задача: пересоздать товары с локальными, а не внешними плейсхолдерами.
- Ключевые моменты:
  - выбирает продавца:
    - сначала по админскому email `bshevelev@mail.ru`, если такой `users` существует;
    - иначе первый доступный профиль из `profiles`;
  - удаляет все `messages`, `cart_items`, `items` (только товары и связанные сущности);
  - создаёт 10 товаров с фиксированными названиями/описаниями и ценами;
  - для каждого товара генерирует 4–8 SVG‑файлов плейсхолдеров в `public/uploads/items` (`svgPlaceholder()`):
    - текст «Фото N» на сером фоне;
    - пути вида `/uploads/items/<filename>.svg` сохраняются в `items.image_url` и `items.image_urls`.
- Таким образом, после запуска:
  - в БД нет ссылок на `/api/placeholder` или внешние домены;
  - все картинки товаров — реальные файлы на диске, доступные через API `/api/uploads/items/[filename]/route.ts`.

### 3.4 Скрипты миграции и диагностики изображений

Из истории в `what's new.md`:

- `scripts/placeholder-to-files.ts`:
  - для товаров с `image_urls`, указывающими на `/api/placeholder`, создаёт реальные файлы в `public/uploads/items/` и обновляет БД;
  - используется для окончательного отказа от сетевого placeholder‑эндпойнта как основного источника картинок.
- `scripts/fix-placeholder-urls.ts`:
  - исправляет некорректные URL в БД (`/images/placeholder.svg`, старые домены, несуществующие пути), переводя их на единый формат.
- `scripts/check-items-db.ts`:
  - диагностирует структуру БД по товарам (наличие и формат `image_url`/`image_urls`, количество записей и т.д.).
- `scripts/diagnose-db.ts`, `docs/DB-MANUAL-INSPECT.md`:
  - общие вспомогательные инструменты для проверки содержимого БД.

### 3.5 Подводные камни с фото (из инцидентов)

Сводка причин и решений (детальный лог см. `what's new.md`):

- **Статика `public/images/placeholder.svg` в production**:
  - Не всегда доступна после сборки/деплоя (особенности Next.js и деплоя через rsync).
  - Решение:
    - добавить API‑маршрут `/api/placeholder` для генерации SVG на лету;
    - использовать `PLACEHOLDER_DATA_URI` в компонентах, чтобы при ошибке загрузки отображался встроенный SVG, не зависящий от сети.
- **Файлы `public/uploads/*` не попадали в production**:
  - Проблема: Next.js production не знает о файлах, появляющихся после `next build`, а rsync/деплой могли перезаписывать или не копировать нужные подкаталоги.
  - Решение:
    - добавить API‑маршруты `/api/uploads/hero/[filename]`, `/api/uploads/items/[filename]`, `/api/uploads/chat/[filename]`, `/api/uploads/agent/[filename]`;
    - настроить rewrites `^/uploads/...` → `^/api/uploads/...` (см. `next.config.ts`);
    - в деплое следить за исключениями/включениями `public/uploads` (см. `scripts/deploy-hoster.ps1`, `docs/HOSTER-RU-DEPLOY.md`).
- **Битые URL (`/images/...`, `picsum.photos`, `placehold.co`) в БД**:
  - После нескольких итераций сидов и миграций часть товаров ссылалась на несуществующие домены или относительные пути без файлов.
  - Решение:
    - использовать `resolveImageUrl()` (`src/lib/image-url.ts`) и `PLACEHOLDER_DATA_URI` для graceful degradation;
    - запустить `placeholder-to-files.ts` / `recreate-items.ts` для переноса данных на единый формат `/uploads/items/...`.

## 4. Деплой, инфраструктура и устойчивость

### 4.1 Стратегия деплоя

- Основной сценарий:
  1. Локальные изменения → обновление `version.json` и `what's new.md` для затронутых компонентов.
  2. `git commit` и `git push origin main`.
  3. Запуск `scripts/deploy-hoster-git.ps1 -Branch main`:
     - на сервере запускается `scripts/deploy-from-git.sh main`;
     - выполняются `npm ci`, `prisma migrate deploy`, `npm run build`, `pm2 restart komissionka agent bot`.
- GitHub Actions:
  - Workflow `deploy.yml` переведён в ручной режим (`workflow_dispatch`) и **не** используется как основной путь деплоя, чтобы:
    - избежать «скрытых» автодеплоев;
    - держать контроль в одном месте — локальном скрипте.
- Резервный деплой:
  - `scripts/deploy-hoster.ps1`:
    - rsync или scp‑загрузка всего репозитория;
    - опции `-Build`, `-Upload`, `-Restart` и `-All`;
    - используется только при сбоях git‑деплоя.

### 4.2 Проблемы prod‑сборки и их решения

**Отсутствие `.next/BUILD_ID` и «вечные» рестарты `next start`**

- Симптомы:
  - `pm2` бесконечно перезапускает приложение;
  - в каталоге `.next` отсутствуют некоторые файлы (`routes-manifest.json`, `BUILD_ID`).
- Типичная причина:
  - `npm run build` падает на этапе проверки типов (`tsc --noEmit`) или на рантайм‑ошибках в API‑маршрутах;
  - пример: параметр цикла без типа (`i` имплиситно `any`) в `src/app/api/admin/data/route.ts`.
- Решение:
  - фикс типов и ошибок в проблемных файлах;
  - деплой через git‑скрипт, который не запускает `next start`, если сборка упала (ошибка видна в логах).

**Undici / llhttp-wasm на Windows и проблемы с fetch**

- Для маршрута `/api/admin/agent/run` раньше использовался `fetch` с `undici`:
  - на Windows это приводило к редким обрывам SSE‑стриминга и нестабильной работе;
  - дополнительно усложняло диагностику ошибок при обращении к локальному агенту.
- Решение:
  - переписать прокси к агенту на низкоуровневый `node:http`:
    - функция `agentRequest()` реализует POST‑запрос к `http://127.0.0.1:AGENT_PORT/run`;
    - SSE‑стрим реализован через `Readable.toWeb(res)` и ручной парсинг `event:`/`data:`;
    - реализованы:
      - двойной ретрай при временных сетевых ошибках;
      - fallback без стрима (получение результата одним блоком);
      - подробные сообщения об ошибках с подсказками по перезапуску агента/порту.

**OOM и память Node.js**

- При сборке или длительной работе приложения на сервере случались OOM‑ошибки.
- Приняты меры (см. `what's new.md`, раздел «OOM fix»):
  - ограничение памяти через `NODE_OPTIONS=--max-old-space-size=512` в `package.json` (`npm start`);
  - `ecosystem.config.cjs` для PM2 с явными лимитами;
  - скрипт `scripts/setup-swap-hoster.ps1` для настройки swap на VPS.

### 4.3 Службы и авто‑перезапуск

- Агент автоматически перезапускает службы (через PM2) перед формированием подвала:
  - логика в `agent/core.ts` (`runRestartThenGetStatus()`), `agent/lib/services-status.ts`, `scripts/check-services-status.ts`;
  - после изменений, требующих перезапуска (`run_command` с `pm2 restart`, миграции и т.д.), агент:
    - выполняет `pm2 restart komissionka agent bot`;
    - получает статус всех трёх сервисов;
    - добавляет его в подвал отчёта.
- В правилах (`AGENTS.md`, `.cursor/rules/versioning.mdc`) зафиксировано:
  - любые изменения кода должны сопровождаться:
    - обновлением версий;
    - корректным подвалом (версии до/после, статус служб, даты ключевых файлов на сервере);
    - фактическим деплоем (а не инструкцией пользователю).

## 5. История ключевых инцидентов (краткий реестр)

Развёрнутая хроника — в `what's new.md`. Ниже — тематическая сводка.

### 5.1 Изображения и uploads

- Многократные проблемы с отображением фото товаров:
  - статика `public/` не попадала на сервер или была недоступна после билдов;
  - внешние сервисы картинок (`picsum.photos`, `placehold.co`) блокировались или давали mixed‑content на HTTPS;
  - URL в БД указывали на `/images/...` вместо `/uploads/...` или `/api/placeholder`.
- Поэтапные решения:
  - собственный API `/api/placeholder` и встроенный `PLACEHOLDER_DATA_URI`;
  - перенос ссылок товаров на реальные файлы в `public/uploads/items` с помощью `placeholder-to-files.ts` и `recreate-items.ts`;
  - API‑обёртки `/api/uploads/*` для статических файлов, появляющихся после сборки;
  - страница и API диагностики (`/diagnose-images`, `/api/diagnose-images`).

### 5.2 Корзина и модель `cart_items`

- Проблема:
  - ранее `cart_items` имела составной первичный ключ по `user_id` и `product_id`, при этом в БД существовало поле `id` без default;
  - Prisma пытался выполнять `upsert` без указания `id`, что на реальной БД приводило к `Null constraint violation` (P2011) при добавлении товара в корзину.
- Решение:
  - привести схему к фактическому состоянию:
    - добавить поле `id` как `@id @default(uuid())`;
    - перенести `@@id([user_id, product_id])` в `@@unique([...])`;
  - после миграции операции с корзиной выполняются корректно.

### 5.3 Агент и коды подтверждения

- Уже упомянутые проблемы:
  - потеря pending‑состояния при рестарте;
  - некорректный TTL;
  - кэширование ответов для кодовых промптов;
  - падения при выполнении подтверждённого плана.
- В итоге:
  - хранилище перенесено на диск с безопасной сериализацией;
  - TTL считается от `lastShownAt`;
  - коды/откаты обходят кэш;
  - любое исключение при выполнении плана оборачивается в ответ пользователю, не убивая процесс агента.

### 5.4 NextAuth, HTTPS и сессии

- Проблемы:
  - некорректная работа cookie и NextAuth без HTTPS на домене;
  - странности с обновлением сессии после логина/логаута.
- Решения:
  - жёсткое требование HTTPS для домена `komissionka92.ru` (см. `docs/manual/app-komissionka.md`, раздел про HTTPS);
  - настройки `NEXTAUTH_URL` и `APP_BASE_URL` на `https://komissionka92.ru` (не `http://` на проде);
  - в UI логина — принудительное `window.location.href` после успешного входа для обновления состояния на клиенте;
  - за nginx: **`AUTH_TRUST_HOST=true`** в `.env` (NextAuth v4 читает эту переменную в `detect-origin`, не `NEXTAUTH_TRUST_HOST`);
  - Let's Encrypt: в SAN сертификата должны быть и `komissionka92.ru`, и `www.komissionka92.ru`, иначе антивирусы/браузеры ругаются на имя сертификата.

### 5.5 Prisma Studio и доступ к БД

- Инцидент:
  - при открытии Prisma Studio в новой вкладке пользователя выбрасывало на главную из‑за того, что профиль ещё не был загружен, и роль считалась `null`.
- Решение:
  - до загрузки профиля не выполнять редирект на `/`, а дождаться результата запроса профиля;
  - Prisma Studio: `/admin/prisma-studio` (вход с ролью admin); ссылка с портала shectory.ru в карточке проекта.

### 5.6 Deploy и скрипты

- Инциденты:
  - старый `deploy-hoster.ps1` использовался как основной путь, а не как резервный;
  - rsync/scp могли перезаписывать `public/uploads`;
  - `deploy-hoster-git.ps1` падал на чистом дереве из‑за некорректной проверки вывода `git status`.
- Решения:
  - зафиксирован единый штатный путь деплоя: git‑скрипт `deploy-hoster-git.ps1` → `scripts/deploy-from-git.sh`;
  - резервный `deploy-hoster.ps1` явно помечен как fallback;
  - скрипты деплоя приведены к устойчивому поведению (корректная проверка статуса Git, аккуратное обращение с `public/uploads`, исключение `.env` и т.п.).
