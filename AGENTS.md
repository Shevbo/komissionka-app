# Инструкции для агентов (Cursor, модель ИИ)

## Месторасположение: СЕРВЕР

**Репозиторий, код, документация, скрипты — на сервере.** VPS 83.69.248.175, каталог `~/komissionka`. Локальная разработка заморожена. Все команды, тестирование, БД, API — на сервере. Выполнение — только через скрипты (`ssh hoster`, `.\scripts\deploy-hoster-git.ps1`, `.\scripts\deploy-hoster.ps1`, `.\scripts\setup-*-hoster.ps1`), без инструкций «выполните вручную».

**Деплой — СТРОГО самостоятельно.** Все агенты обязаны сами деплоить свои изменения. Пользователь только наблюдает статусы в админке; от пользователя ничего не просят (ни коммит, ни «решите, что включить», ни деплой). Перед деплоем дерево должно быть чистым: все свои изменения коммитишь и пушишь сам, затем запускаешь `.\scripts\deploy-hoster-git.ps1`. Не предлагать пользователю «подготовить коммит за вас» или «сначала сделайте коммит» — делай коммит и push сам. Резерв при полной недоступности очереди: только `.\scripts\deploy-hoster.ps1 -Upload -Restart` (scp).

## Версионность

При изменении кода — правила из `docs/VERSIONING-RULES.md` и `.cursor/rules/versioning.mdc`:

1. X+1 — >30% core; Y+1 — <30% core; Z+1 — баг. При X+1/Y+1 сброс Z в 0.
2. **ЗАПРЕЩЕНО** менять версии по запросу пользователя. Только при фактических изменениях.
3. what's new.md — блок UPDATE с количеством заменённых строк и строками в core.
4. Подвал отчёта (обязательно) — **две строки**: «Версии до: …» и «Версии после: …» (см. `.cursor/rules/versioning.mdc` §5).

## Распознавание задания и подвал отчёта (обязательно)

**Перед версионированием** явно определи: фича (Y+1) или баг (Z+1). Новая функциональность — фича, сбой/ошибка — баг. Это указывай в **теле** отчёта при необходимости, не в подвале.

Подвал — **только две строки**:

```
Версии до: app v..., agent v..., tgbot v...
Версии после: app v..., agent v..., tgbot v...
```

Без этих двух строк отчёт о внесённых изменениях считается неполным.

## Деплой (всегда автоматом, без исключений)

После любых изменений кода — всегда автоматом: коммит → push → деплой `.\scripts\deploy-hoster-git.ps1`. Не спрашивать «закоммитить?» / «задеплоить?» — делать сразу. Пользователь не деплоит — только смотрит «Деплой» в админке. Если очередь недоступна — скрипт завершится с ошибкой; резерв только scp: `.\scripts\deploy-hoster.ps1 -Upload -Restart`.

**Как деплоить (для агента в другом чате):** пошаговая инструкция — [docs/manual/DEPLOY-AGENT.md](docs/manual/DEPLOY-AGENT.md). Кратко: из корня проекта `.\scripts\deploy-hoster-git.ps1` (прод), для тестовой среды `.\scripts\deploy-hoster-git.ps1 -Env test1`. Вкладка «Деплой» в админке (https://komissionka92.ru/admin) показывает среды, очередь и журнал; пустая очередь после выполнения операций — норма, активность смотри в «Журнал операций». При сбое очереди — резерв только scp: `.\scripts\deploy-hoster.ps1 -Upload -Restart`.
