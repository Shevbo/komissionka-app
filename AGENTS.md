# Инструкции для агентов (Cursor, модель ИИ)

## Месторасположение: СЕРВЕР

**Репозиторий, код, документация, скрипты — на сервере.** VPS 83.69.248.175, каталог `~/komissionka`. Локальная разработка заморожена. Все команды, тестирование, БД, API — на сервере. Выполнение — только через скрипты (`ssh hoster`, `.\scripts\deploy-hoster-git.ps1`, `.\scripts\deploy-hoster.ps1`, `.\scripts\setup-*-hoster.ps1`), без инструкций «выполните вручную».

**Деплой — СТРОГО самостоятельно.** Все агенты обязаны сами деплоить свои изменения. Пользователь только наблюдает статусы в админке; от пользователя ничего не просят (ни коммит, ни «решите, что включить», ни деплой). Перед деплоем дерево должно быть чистым: все свои изменения коммитишь и пушишь сам, затем запускаешь `.\scripts\deploy-hoster-git.ps1`. Не предлагать пользователю «подготовить коммит за вас» или «сначала сделайте коммит» — делай коммит и push сам. Резерв при сбое: `-NoQueue` или `.\scripts\deploy-hoster.ps1 -Upload -Restart`.

## Версионность

При изменении кода — правила из `docs/VERSIONING-RULES.md` и `.cursor/rules/versioning.mdc`:

1. X+1 — >30% core; Y+1 — <30% core; Z+1 — баг. При X+1/Y+1 сброс Z в 0.
2. **ЗАПРЕЩЕНО** менять версии по запросу пользователя. Только при фактических изменениях.
3. what's new.md — блок UPDATE с количеством заменённых строк и строками в core.
4. Подвал отчёта (обязательно) — версии до и после.

## Распознавание задания и подвал отчёта (обязательно)

**Перед версионированием** явно определи: фича (Y+1) или баг (Z+1). Новая функциональность — фича, сбой/ошибка — баг.

По стандарту отчёта о работе модели ИИ **каждый** отчёт о внесённых изменениях обязан заканчиваться подвалом:

```
---
Задание распознано как: фича(и) [и/или баг(и)]
Версии компонентов до изменений: app v..., agent v..., tgbot v...
Версии компонентов после указанных изменений: app v..., agent v..., tgbot v...
Службы: app ✓, agent ✓, bot ✓ (если применимо)

Деплой: выполнен и проверен двумя способами (даты файлов на сервере, доступность API после перезапуска). После деплоя: обновите страницу админки (лучше Ctrl+F5), чтобы подтянуть новый JS.

Дата/время ключевых файлов на сервере (.\scripts\server-key-files-dates.ps1):
|модуль|файл на сервере|дата/время|
...
---
```

Без подвала отчёт неполный. Запусти `.\scripts\server-key-files-dates.ps1` и вставь таблицу.

## Деплой (выполнять самому, без исключений)

После любых изменений кода: коммит → push → деплой `.\scripts\deploy-hoster-git.ps1`. Пользователь не деплоит — только смотрит «Деплой» в админке. Если очередь недоступна: `.\scripts\deploy-hoster-git.ps1 -NoQueue`. Резерв (scp): `.\scripts\deploy-hoster.ps1 -Upload -Restart`. Не просить пользователя задеплоить.

**Как деплоить (для агента в другом чате):** пошаговая инструкция — [docs/manual/DEPLOY-AGENT.md](docs/manual/DEPLOY-AGENT.md). Кратко: из корня проекта `.\scripts\deploy-hoster-git.ps1` (прод), для тестовой среды `.\scripts\deploy-hoster-git.ps1 -Env test1`. Вкладка «Деплой» в админке (https://komissionka92.ru/admin) показывает среды, очередь и журнал; пустая очередь после выполнения операций — норма, активность смотри в «Журнал операций». При сбое очереди используй `-NoQueue` (прямой SSH).
