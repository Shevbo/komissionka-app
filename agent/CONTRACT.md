# Контракт агента (вход — выход)

## Вход

- **prompt** (обязательно) — один текстовый промпт, строка.
- **Параметры** (опционально, на последующих этапах):
  - `maxOutputLength` — максимальная длина ответа (символов);
  - `timeoutMs` — таймаут на выполнение запроса (мс).

## Выход

- **result** — один текстовый ответ (финальная выдача для пользователя).
- **debugLog** (опционально) — структурированный лог шагов, только для отладки и логов, не отдаётся пользователю.

## Способ вызова (на данном этапе)

Выбран **Вариант 2: агент как процесс**.

- **stdin** — тело запроса (промпт). Сервис или бот запускает процесс, передаёт промпт в stdin и читает ответ из stdout.
- **stdout** — финальный ответ (только текст результата).

Примеры:

```bash
# Из корня проекта
echo "Опиши структуру проекта" | npm run agent:start
```

В коде вызывается `runAgent(prompt, options?)` → `Promise<{ result: string; debugLog?: unknown }>`.

## Альтернативы (для следующих этапов)

- **Вариант 1 (HTTP):** POST с телом `{ "prompt": "..." }`, ответ `{ "result": "..." }` или `{ "error": "..." }`.
- **Вариант 3 (MCP):** агент как MCP-сервер; вызов через протокол MCP.

## Заглушка (Этап 1.2)

На любой промпт агент возвращает фиксированную строку: `Agent stub`. Интерфейс готов для подключения ядра агента на Этапе 3.

## Окружение и конфигурация (Этап 1.3)

Источники конфигурации (приоритет: переменные окружения > `agent/config.json`):

1. **Переменные окружения** с префиксом `AGENT_` (можно задать в `.env` в корне проекта или при запуске).
2. **Файл** `agent/config.json` (опционально) — те же ключи, без префикса (см. пример ниже).

### Список переменных окружения

| Переменная | Описание | По умолчанию |
|------------|----------|--------------|
| `AGENT_ROOT` | Рабочая директория агента (корень репозитория). Все пути к файлам — относительно неё. | `process.cwd()` при запуске |
| `AGENT_DEBUG` | Включить отладочный лог в ответе (`debugLog`). Значения: `1`, `true`, `yes` | — |
| `AGENT_LLM_API_KEY` | API-ключ облачной LLM (для этапов 3+) | — |
| `AGENT_LLM_MODEL` | Модель LLM (например, `gpt-4o`, Ollama: `llama3`) | — |
| `AGENT_LLM_BASE_URL` | Базовый URL API (Ollama: `http://localhost:11434/v1`; OpenRouter: `https://openrouter.ai/api/v1`) | — |
| `AGENT_CONTEXT_FILE` | Путь к файлу с контекстом проекта (относительно корня), например `docs/AGENT-CONTEXT.md`. Содержимое добавляется в системный промпт («опыт» агента по проекту). | — |
| `AGENT_TIMEOUT_MS` | Таймаут одного запроса к агенту (мс) | `120000` |
| `AGENT_MAX_OUTPUT_LENGTH` | Максимальная длина ответа (символов) | `8192` |
| `AGENT_PORT` | Порт HTTP-сервера (режим «долгоживущий процесс», Вариант A) | `3140` |
| `AGENT_API_KEY` | Секрет для вызова API (Authorization: Bearer или X-API-Key). Если задан — без ключа запросы отклоняются. | — |
| `AGENT_PROXY` | URL прокси для запросов к LLM (например, `https://user:pass@host:port`). Используется также `AGENT_HTTPS_PROXY` / `AGENT_HTTP_PROXY`. Нужен для обхода региональных ограничений (например, Google AI). | — |
| `AGENT_PROXY_CONNECT_TIMEOUT_MS` | Таймаут подключения к прокси (мс). При «Connect Timeout» увеличьте или проверьте доступность прокси. | `30000` |

### Передача конфигурации при запуске

- **Локально:** в корне проекта создайте или дополните `.env` (переменные с префиксом `AGENT_`). При запуске `npm run agent:start` скрипт загружает `.env` из текущей рабочей директории (корень проекта). Либо задайте переменные в shell: `AGENT_DEBUG=1 npm run agent:start`.
- **На сервере:** те же переменные передайте через окружение процесса (systemd `Environment=`, PM2 `env`, Docker `-e` или `env_file`). Либо положите `.env` в каталог запуска и запускайте агент из корня репозитория.

### Пример `agent/config.json` (опционально)

```json
{
  "root": "/path/to/komissionka",
  "debug": false,
  "llmApiKey": "",
  "llmBaseUrl": "",
  "llmModel": "gpt-4o",
  "timeoutMs": 120000,
  "maxOutputLength": 8192
}
```

Секреты (API-ключи) предпочтительно задавать переменными окружения, а не в `config.json`.

### Пример: Claude 4.6 Sonnet (через OpenRouter)

Текущий клиент агента совместим с OpenAI API. Чтобы использовать **Claude 4.6 Sonnet**, удобно взять [OpenRouter](https://openrouter.ai): один endpoint и выбор модели по имени.

В `.env` в корне проекта:

```env
AGENT_LLM_API_KEY=sk-or-v1-...ваш_ключ_OpenRouter...
AGENT_LLM_BASE_URL=https://openrouter.ai/api/v1
AGENT_LLM_MODEL=anthropic/claude-sonnet-4.6
```

Ключ берётся в [OpenRouter Keys](https://openrouter.ai/keys). Модель `anthropic/claude-sonnet-4.6` — Claude Sonnet 4.6. Дальше запуск как обычно: `echo "промпт" | npm run agent:start`.

(Если нужен именно нативный API Anthropic, потребуется отдельный адаптер в коде, т.к. формат запросов у Anthropic другой.)

### Пример: Gemini 3.1 Pro (Google AI)

Google предоставляет [OpenAI-совместимый endpoint](https://ai.google.dev/gemini-api/docs/openai) для Gemini. Текущий клиент агента подходит без изменений.

В `.env` в корне проекта:

```env
AGENT_LLM_API_KEY=ваш_ключ_из_Google_AI_Studio
AGENT_LLM_BASE_URL=https://generativelanguage.googleapis.com/v1beta/openai
AGENT_LLM_MODEL=gemini-3.1-pro-preview
```

Ключ (формат `AIza...`) создаётся в [Google AI Studio](https://aistudio.google.com/apikey). Модель `gemini-3.1-pro-preview` — Gemini 3.1 Pro Preview. Запуск: `echo "промпт" | npm run agent:start` или `npm run agent:serve` и POST /run.

**Важно:** не коммитьте `.env` с ключом в репозиторий (`.env` должен быть в `.gitignore`).

### Прокси для запросов к LLM

Если провайдер LLM (например, Google AI) недоступен из вашего региона, задайте прокси — все запросы к API пойдут через него:

```env
AGENT_PROXY=https://адрес_прокси:порт
```

Поддерживаются схемы **HTTP** и **HTTPS** прокси (`http://...` или `https://...`). С учётными данными: `https://user:password@host:port`. Альтернативно: `AGENT_HTTP_PROXY`, `AGENT_HTTPS_PROXY`. После добавления переменной перезапустите агент (`npm run agent:serve`).

### Контекст проекта («опыт» агента)

Чтобы агент знал текущее состояние проекта (структура, что уже сделано, соглашения), задайте файл с текстом и подставьте его в системный промпт:

```env
AGENT_CONTEXT_FILE=docs/AGENT-CONTEXT.md
```

Создайте в репозитории `docs/AGENT-CONTEXT.md` и опишите там кратко: что за проект, основные каталоги, какие этапы/фичи уже реализованы, куда смотреть в коде. Содержимое этого файла добавляется к системному промпту при каждом запросе. Шаблон и пример — в `docs/AGENT-CONTEXT.md`.

## Инструменты агента (Этап 2)

Набор tools для чтения и поиска по проекту (и опционально — выполнения разрешённых команд).

### 2.1 Базовые инструменты чтения

- **read_file(path)** — прочитать файл по пути относительно корня. Разрешены каталоги: `src/`, `prisma/`, `docs/`, `agent/`, `public/` и корневые конфиги (`package.json`, `tsconfig.json` и т.д.). Пути вне репозитория и вне allowlist отклоняются.
- **list_dir(path)** — список файлов и папок в каталоге (без рекурсии). Путь по умолчанию — корень.
- **find_files(pattern)** — поиск файлов по маске. Поддержка `*` и `**`. Примеры: `**/*.ts`, `prisma/*.prisma`, `src/**/*.tsx`.

### 2.2 Поиск по содержимому

- **grep(search_string, path?, options?)** — поиск по тексту в файлах (подстрока или регулярное выражение). Опции: `context_lines`, `use_regex`, `filePattern`. Возвращает совпадения с контекстом (строки до и после).

### 2.3 Безопасное выполнение команд

- **run_command(command, cwd?, timeout?)** — выполнить команду из белого списка: `npx prisma generate/validate/format`, `npm run build/lint/dev`, `npx tsc --noEmit`. Секреты из `.env` не передаются в окружение процесса; таймаут задаётся конфигом или параметром.

### 2.4 Описание для LLM

Массив **TOOLS_FOR_LLM** в `agent/tools/index.ts` содержит описания в формате OpenAI function calling (имя, описание, параметры с типами). Функция **executeTool(name, args)** выполняет вызов и возвращает текстовый результат для вставки в контекст.

### Тесты

Запуск из корня проекта: `npx tsx agent/tools/tools.test.ts` или из каталога agent: `npm run test`.

## Ядро агента (Этап 3)

Цикл «промпт → LLM → при необходимости вызов инструментов → финальный ответ».

### 3.1 Клиент LLM

Модуль `agent/llm/client.ts`: `request(messages, tools?, options) => response`. Совместим с OpenAI API и с Ollama (задать `AGENT_LLM_BASE_URL=http://localhost:11434/v1`, модель например `llama3`). Системный промпт и ответы пользователя передаются в `messages`; при наличии `tool_calls` в ответе они выполняются, результаты добавляются в диалог, запрос повторяется.

### 3.2 Системный промпт

Шаблон в `agent/llm/system-prompt.ts`: агент — помощник по проекту «Комиссионка», стек (Next.js, Prisma, PostgreSQL), перечень инструментов и требование возвращать краткий понятный текст.

### 3.3 Цикл рассуждений

В `agent/core.ts`: цикл вызовов LLM с ограничением (до 8 итераций инструментов). При ответе с `tool_calls` — выполнение инструментов, добавление результатов в `messages`, повтор запроса; при текстовом ответе — возврат пользователю (с обрезкой по `maxOutputLength`).

### 3.4 Ошибки и таймаут

Таймаут на весь запрос задаётся `AGENT_TIMEOUT_MS` (или опцией `timeoutMs` в `runAgent`). При ошибке LLM, инструмента или таймауте возвращается короткое сообщение пользователю; детали пишутся в stderr (`console.error`).

## Запуск и жизненный цикл (Этап 4)

Выбран **Вариант A — долгоживущий процесс**: агент поднимается как HTTP-сервер и обрабатывает запросы по мере поступления.

- **Запуск:** `npm run agent:serve` из корня проекта. Сервер слушает порт из `AGENT_PORT` (по умолчанию 3140).
- **Контракт:** POST /run с телом `{ "prompt": "..." }` → ответ `{ "result": "..." }` или `{ "error": "..." }`. Опционально: GET /health → `{ "status": "ok" }`.
- **Защита:** при заданном `AGENT_API_KEY` запросы без заголовка `Authorization: Bearer <key>` или `X-API-Key: <key>` отклоняются (401).

Режим «процесс на запрос» (stdin/stdout) по-прежнему доступен: `echo "промпт" | npm run agent:start`.

Подробно: **docs/AGENT-RUN.md** — локальный запуск, проверка через curl, запуск на сервере (PM2, systemd).
