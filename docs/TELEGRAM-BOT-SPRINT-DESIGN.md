# Телеграм-бот «Спринт Комиссионки» — проектирование и дорожная карта

## 1. Назначение сервиса

Бот даёт администраторам «Комиссионки» возможность отправлять **промпты из Telegram** и получать в ответ **результат выполнения**, как если бы промпт был введён в чате Cursor в контексте проекта «Комиссионка». Доступ только у администраторов приложения.

---

## 2. Требования (сводка)

| № | Требование | Реализация в проекте |
|---|------------|----------------------|
| 1 | Доступ только для администраторов «Комиссионки» | Проверка по привязке Telegram → профиль с `role = admin` в БД проекта |
| 2 | Получать промпт из чата Telegram | Telegram Bot API: обработка текстовых сообщений |
| 3 | Выполнять промпт как в чате Cursor проекта «Комиссионка» | Сервис «исполнителя»: вызов AI-агента с контекстом репозитория (см. варианты ниже) |
| 4 | Возвращать финальную выдачу в чат Telegram | Отправка ответа ботом в тот же чат (с учётом лимита длины сообщения) |

---

## 3. Архитектура сервиса

```

        │                                        │    (admin по telegram_id)
        │                                        │ 2) Вызов «исполнителя»
        │                                        ▼
        │                               ┌──────────────────┐
        │                               │  Executor        │
        │                               │  (см. варианты   │
        │                               │   ниже)          │
        └──────────────────────────────┴────────┬─────────┘
                                                 │
                    ┌────────────────────────────┼────────────────────────────┐
                    │                            │                            │
                    ▼                            ▼                            ▼
           ┌────────────────┐          ┌────────────────┐          ┌────────────────┐
           │ Вариант A:     │          │ Вариант B:     │          │ Вариант C:     │
```

### 3.1 Компоненты

| Компонент | Назначение |
|-----------|------------|
| **Telegram Bot** | Приём сообщений, проверка прав, вызов исполнителя, отправка ответа в Telegram. Может быть отдельным сервисом (отдельный процесс/контейнер) или набором API-маршрутов в текущем Next.js (webhook). |
| **Проверка прав** | По `telegram_id` (или username) определить пользователя и проверить в БД «Комиссионки», что у соответствующего `profiles` поле `role = 'admin'`. Для этого в БД нужна привязка «профиль ↔ Telegram». |
| **Executor (исполнитель промпта)** | Принимает текст промпта, возвращает итоговый текст ответа. Реализация — один из вариантов A/B/C ниже. |

### 3.2 Варианты реализации «исполнителя» (п.3–4 требований)

- **Вариант A — Cursor API / Cursor CLI**  
  Если у Cursor есть публичный API или CLI, принимающий промпт и путь к проекту и возвращающий ответ агента — сервис бота вызывает его и отдаёт вывод в Telegram.  
  *Ограничение:* на момент проектирования публичный «Cursor Chat API» для сторонних ботов может быть недоступен; необходимо уточнять актуальные возможности Cursor (документация, поддержка).

- **Вариант B — OpenAI (или другой AI) API + контекст проекта**  
  Сервис формирует запрос к API (OpenAI, Anthropic и т.п.): системный промпт с указанием «проект Комиссионка, Next.js, Prisma», пользовательский промпт из Telegram. Ответ модели возвращается в Telegram.  
  *Плюсы:* реализуемо сразу, предсказуемая стоимость. *Минусы:* нет полного контекста кодовой базы, как в Cursor; при необходимости можно добавлять выбранные файлы/сводки в контекст.

- **Вариант C — Локальный агент / MCP в репозитории**  
  В репозитории «Комиссионка» запускается агент (например, через MCP task или свой воркер), который получает промпт, выполняет его в контексте проекта (чтение файлов, запуск кода, вызов инструментов) и возвращает текстовый результат. Бот вызывает этого агента по HTTP или очереди.  
  *Плюсы:* максимально близко к «выполнению в Cursor». *Минусы:* сложнее в настройке и эксплуатации, нужен постоянно работающий процесс или триггер запуска.  
  **Пошаговый план реализации Варианта C:** см. [TELEGRAM-BOT-VARIANT-C-STAGES.md](TELEGRAM-BOT-VARIANT-C-STAGES.md).

**Рекомендация для старта:** реализовать **Вариант B** (AI API с контекстом проекта), параллельно уточнять возможность **Варианта A** (Cursor). При появлении доступа к Cursor API — добавить режим «Cursor» и при желании переключить бота на него.

## 4. Данные и интеграция с «Комиссионкой»

### 4.1 Привязка администратора к Telegram

Нужно хранить идентификатор Telegram пользователя для профилей-админов.

**Вариант 1 (рекомендуется):** поле в `profiles`:

- `telegram_id` (BigInt или String) — уникальный ID пользователя в Telegram (получает бот из `message.from.id`).
- Опционально: `telegram_username` (String) — для отображения и ручной привязки по @username.

**Вариант 2:** отдельная таблица `admin_telegram_bindings (profile_id, telegram_id, created_at)` — если не хотите трогать `profiles`.

Проверка прав при каждом сообщении боту:

1. Взять `telegram_id` из `message.from.id`.
2. Найти профиль с `telegram_id = ...` и `role = 'admin'` (или по таблице привязок).
3. Если не найден — ответ бота: «Доступ только для администраторов Комиссионки. Привяжите Telegram в настройках админки.»

Привязку Telegram к аккаунту админа можно сделать в админ-панели «Комиссионки»: админ входит, нажимает «Привязать Telegram», бот выдаёт одноразовый код или ссылку; ввод кода в боте подтверждает привязку и сохраняет `telegram_id` в БД.

### 4.2 Схема БД (минимальное изменение)

Добавить в модель `profiles` (или в новую таблицу):

```prisma
// В profiles добавить:
telegram_id   String?  @unique  // Telegram user id (message.from.id)
telegram_username String?       // @username для отображения
```

Миграция: `npx prisma migrate dev --name add_telegram_to_profiles`.

## 5. Безопасность

- **Секреты:** токен бота (`TELEGRAM_BOT_TOKEN`), при варианте B — ключ AI API. Хранить в переменных окружения, не в коде.
- **Вебхук:** если бот по вебхуку — только HTTPS, проверка подписи Telegram (secret_token).
- **Ограничение доступа:** только привязанные админы; все остальные получают отказ без выполнения промпта.
- **Лимиты:** ограничить длину промпта и частоту запросов от одного пользователя (rate limit), чтобы снизить риски и расходы на AI API.
- **Логирование:** логировать только факт вызова (user_id, длина промпта), без хранения полного текста промптов в логах, если они чувствительные.

---

## 6. Ограничения Telegram

- Длина одного сообщения бота — до **4096** символов. Если ответ исполнителя длиннее — разбить на несколько сообщений или отправить файлом (например, .txt).
- Таймаут ответа: Telegram ждёт ответ на обновление (при long polling) ограниченное время; при долгом выполнении промпта — сначала отправить «Обрабатываю…», затем отправить результат отдельным сообщением (например, через очередь или фоновую задачу).

## 9. Пример сценария использования

1. Админ открывает админ-панель «Комиссионки», нажимает «Привязать Telegram», видит код «КОМ-12345».
2. В Telegram пишет боту: «Привязать КОМ-12345». Бот проверяет код, находит профиль с `role=admin`, сохраняет `telegram_id`, отвечает: «Telegram привязан. Можете отправлять промпты.»
3. Админ пишет: «Напиши, какие API-маршруты есть в проекте». Бот передаёт промпт в исполнитель (AI API с контекстом проекта), получает ответ, отправляет его в чат (при необходимости частями или файлом).
4. Админ видит в Telegram список маршрутов и краткое описание — как если бы спросил то же самое в чате Cursor в проекте «Комиссионка».
