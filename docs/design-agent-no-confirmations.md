# Проект: Агент без внешних подтверждений («да»/коды)

## 1. Цель

Перестроить архитектуру агента так, чтобы:

- не было внешнего шага подтверждения («да», 4‑значный код) через `/api/admin/agent/run`;
- все уточнения, опросы и подтверждения выполнялись внутри диалога с моделью ИИ (паттерн Cursor‑агента);
- агент имел широкие права на изменения в коде, БД и окружении, но с жёсткими ограничениями (например, защита пользователя `bshevelev@mail.ru`).

Исходная мотивация: эпический баг «Не удалось отправить запрос к агенту.» после нажатия «да» из‑за сложного механизма подтверждений и конфликтов по порту 3140.

## 2. Текущее состояние (высокоуровнево)

- Агент:
  - имеет механику «pending approval» и подтверждения кода/«да» в `agent/core.ts`, `agent/contract.ts`, `agent/serve.ts`;
  - может попадать в нестабильное состояние (EADDRINUSE на порту 3140) как раз на шаге подтверждения.
- Web/API:
  - `/api/admin/agent/run` умеет работать со стримингом шагов и логами, но содержит текстовые и поведенческие рудименты, связанные с подтверждением (коды, «да», подсказки).
- Телеграм‑бот:
  - вызывает агента напрямую по порту `AGENT_PORT`, сам по себе логики подтверждений не содержит.

## 3. Новый подход (как у Cursor‑агента)

Ключевая идея: агент сам ведёт диалог‑интервью, планирует и исполняет задачу внутри одного сеанса LLM, **без отдельного технического шага подтверждения через `/run`**.

### 3.1. Поведение модели по шагам

1. **Уточнение задачи**
   - Модель читает входной промпт пользователя.
   - Задаёт уточняющие вопросы, пока не прояснит:
     - цель (что именно нужно получить),
     - границы системы (какие модули/файлы/данные можно трогать),
     - ограничения (по времени, по объёму изменений и т.п.).

2. **Планирование**
   - Модель формирует краткий план действий (список шагов).
   - Может спрашивать пользователя, устраивает ли такой план (Y/N, варианты поправок) — это **обычный текстовый диалог**, а не внешний слой подтверждения.
   - Модель **не имеет права блокировать выполнение задачи требованием единственного глобального подтверждения**; любой осмысленный ответ пользователя считается продолжением диалога, а не отдельной «командой на запуск плана».

3. **Пошаговая реализация**
   - Для каждого шага модель:
     - выбирает необходимые инструменты (чтение файлов, правки, Prisma, команды);
     - выполняет действия;
     - коротко отчитывается (какие файлы изменены, какие таблицы затронуты, какие команды выполнены).
   - При необходимости может уточнять у пользователя детали по ходу (всё внутри одного диалога).

4. **Финальный отчёт**
   - Модель описывает, что сделано, какие версии изменены, какие инструкции/документы обновлены.

### 3.2. Технически: системный промпт

В `agent/llm/system-prompt.ts`:

- удалить/ослабить инструкции про «запрос кода/да»;
- добавить секцию:
  - про обязательное уточнение задачи и план;
  - про пошаговое исполнение;
  - про аккуратное использование инструментов и соблюдение ограничений по правам (см. раздел 4).

Промпт должен быть вынесен в один явный модуль и версионироваться (например, поле `prompt_version`).

## 4. Политика прав агента

### 4.1. Разрешено

- **Файлы**
  - Чтение, создание, изменение и удаление файлов в:
    - `src/`
    - `prisma/`
    - `agent/`
    - `telegram-bot/`
    - `scripts/`
    - `docs/`
    - корневые конфиги проекта
  - Любые операции с путями **вне корня проекта** `/home/ubuntu/komissionka` запрещены.

- **База данных (Prisma)**
  - Любые операции чтения, создания, обновления и удаления записей, **кроме**:
    - запрет на удаление или изменение пользователя, у которого текущий email равен `bshevelev@mail.ru`, через `prisma.profiles.delete`, `deleteMany`, каскадные операции;
    - запрет на использование `deleteMany` по ключевым таблицам (profiles/users и т.п.) **без явного `where`‑фильтра**; массовые удаления допускаются только с фильтром, сформулированным в явной задаче пользователя.

- **Команды / окружение**
  - Разрешённые команды (по аналогии с уже существующим белым списком из системного промпта для агента):
    - `npx prisma generate`, `npx prisma validate`, `npx prisma migrate dev/deploy/reset` (с параметрами);
    - `npm run build`, `npm run lint`, `npm run dev` (для разработки);
    - `npx tsc --noEmit`;
    - `curl` к публичному API приложения и админ‑API (через `X-Agent-API-Key`);
    - `npx tsx scripts/*.ts` (скрипты из каталога `scripts/`);
    - деплой‑скрипты уровня `scripts/deploy-from-git.sh`, `scripts/deploy-hoster-git.ps1` (с учётом уже заданной политики).
  - Фактический белый список команд должен задаваться **в одном месте** (модуль обёртки над `run_command` / конфиг агента) и служить источником истины; этот документ описывает принципы.

### 4.2. Запрещено

- Любые операции, ведущие к:
  - удалению или повреждению системных каталогов (`/`, `/home`, `/var`, и т.п.);
  - удалению/порче git‑репозитория (`.git`);
  - уничтожению БД (сырые `DROP DATABASE`, `TRUNCATE` без прямого задания).
- Явно в коде агента:
  - добавить обёртки вокруг:
    - Prisma‑удалений (guard на `bshevelev@mail.ru`);
    - `run_command`/`ssh` — фильтрация известных опасных шаблонов (`rm -rf /`, `:(){ :|:& };:` и т.п.).

## 5. Удаление старой логики подтверждений

### 5.1. В агенте

- `agent/core.ts` / `agent/contract.ts` / `agent/serve.ts`:
  - удалить:
    - хранение «pending approval»;
    - запросы 4‑значного кода;
    - ветви логики, которые:
      - строят план и ждут отдельное сообщение «да/ok»;
      - хранят состояние между двумя HTTP‑вызовами `/run`.
  - убедиться, что:
    - `runAgent` или аналогичный entrypoint принимает задачу,
    - делает опрос/планирование/исполнение **внутри одного вызова** (каскад инструментов),
    - возвращает финальный результат и шаги (steps/logId), но не требует повторного вызова.

### 5.2. В `/api/admin/agent/run`

- Упростить маршрут:
  - оставить только:
    - приём `prompt/history/mode/...`;
    - стриминг шагов и результата;
    - обработку ошибок сети/порта;
  - удалить:
    - любые тексты/обработку, привязанные к «коду подтверждения», «повторите 4 цифры», «отправьте да для выполнения плана».

## 6. Фронт (админка)

### 6.1. Вкладка «Комиссионка AI»

- Проверить `src/app/admin/page.tsx`:
  - нет ли UI‑состояний, завязанных на «ожидание подтверждения»;
  - убрать/обновить надписи типа:
    - «Отправьте второе сообщение „да“ для выполнения плана».
- Обновить UX‑тексты:
  - объяснить, что агент:
    - сначала задаёт уточняющие вопросы и предлагает план;
    - затем сам приступает к реализации;
    - пользователь взаимодействует с ним как с Cursor‑агентом (диалог, а не «план + отдельное да»).

### 6.2. Ошибки и диагностика

- Сохранить:
  - `/api/admin/agent/diagnose`;
  - «ДАМП СОСТОЯНИЯ» как основной инструмент диагностики;
  - подсказки в UI типа «нажмите „ДАМП СОСТОЯНИЯ“ при ошибке».
- Убедиться, что тексты ошибок больше не ссылаются на коды/подтверждение.

## 7. Телеграм‑бот (синхронизация)

- Проверить `telegram-bot/bot.ts`:
  - вызов к агенту по `AGENT_PORT` и API `/run?stream=1`;
  - отсутствие ожидания внешнего шага подтверждения.
- При необходимости:
  - скорректировать сообщения бота, если там явно упоминаются коды/подтверждение (ожидаемо — нет).

## 8. Версионирование и документация

- Версии:
  - `agent`: как минимум `1.6.0` (новая архитектура работы с задачами и подтверждениями).
  - `app`: +патч/минор, если API/админка меняются.
  - `tgbot`: без изменений, если логика бота только синхронизируется; минор, если вносятся функциональные изменения.
- Документация:
  - обновить `what's new.md` (корневой и, при необходимости, модульные);
  - обновить `docs/manual/*`:
    - описать новый режим работы агента;
    - перечислить разрешённые/запрещённые действия;
    - описать сценарий взаимодействия администратора с агентом.

## 9. Этапы реализации

1. Этап 1–2: инвентаризация кода подтверждений, проект нового системного промпта.
2. Этап 3: формализация и внедрение политики прав (prompt + guards в коде).
3. Этап 4: удаление старой логики подтверждений в агенте и API.
4. Этап 5: адаптация UI админки.
5. Этап 6: проверка и, при необходимости, правки телеграм‑бота.
6. Этап 7: обновление версий, документации, деплой.
7. Этап 8: интеграционные проверки с админкой и телеграм‑ботом — убедиться, что новый формат шагов/ответов агента не ломает:
   - визуализацию хода выполнения в AI‑чате админки;
   - кнопку «Последний ход» и связанные с ней сценарии в телеграм‑боте.
