## 1. Цели и границы системы

Цель: создать **универсальный каталог тест‑кейсов** для всех модулей проекта «Комиссионка» (web‑приложение, агент к модели ИИ, Telegram‑бот, БД, фоновые воркеры и т.п.), который:

- хранит формализованные сценарии тестирования;
- позволяет запускать тесты:
  - из UI админки (отдельная вкладка);
  - из чата с ИИ‑агентом по текстовой команде («ВЫПОЛНИ СЦЕНАРИЙ ТЕСТИРОВАНИЯ №N»);
- собирает и хранит **полную историю прогонов**:
  - шаги выполнения;
  - диалог с моделью (если используется ИИ);
  - технические логи, дампы состояния, reasoning log;
  - результат сравнения с ожиданиями.

Система должна быть **нейтральной к типу теста**:

- тесты ИИ‑агента и админки;
- тесты API (`/api/*`);
- тесты Telegram‑бота;
- тесты фоновых скриптов/деплоя.

## 2. Объектная модель

### 2.1. Тестируемые модули (TestModule)

Единый справочник модулей системы:

- Web‑приложение (`app`);
- Агент к модели ИИ (`agent`);
- Telegram‑бот (`tgbot`);
- PWA (если выделим отдельно);
- Прочие сервисы (например, deploy‑worker).

В БД это может быть либо отдельная таблица `test_modules`, либо зашитый enum. Минимальный состав полей:

- `id` (string, pk, например `app`, `agent`, `tgbot`);
- `name` (string) — человекочитаемое имя;
- `description` (string, nullable) — краткое описание;
- `active` (boolean).

### 2.2. Тест‑кейсы (TestCase)

Универсальная сущность, описывающая сценарий тестирования.

Ключевые поля:

- `id` (uuid) — внутренний идентификатор.
- `number` (int, unique) — «№ ТЕСТ КЕЙСА» (человекочитаемый номер, по которому общаемся в чате).
- `moduleId` (fk → TestModule.id) — основной проверяемый модуль (web/app/agent/tgbot/...).
- `title` (string) — краткое название («Ход выполнения всегда актуален»).
- `description` (text, markdown) — подробное текстовое описание сценария.
- `kind` (enum) — тип теста:
  - `manual-guided` — выполняется вручную человеком по инструкции, но результат фиксируется в системе;
  - `semi-automatic` — действия делает ИИ‑агент/скрипты, часть шагов может требовать подтверждения;
  - `automatic` — полностью автоматический запуск.
- `scope` (enum) — область:
  - `ui`;
  - `api`;
  - `agent`;
  - `telegram`;
  - `db`;
  - `infra`.
- `promptTemplate` (text, nullable) — шаблон промпта для ИИ (если тест запускает модель). Использует плейсхолдеры параметров.
- `parameters` (json) — универсальный словарь параметров теста:
  - для ИИ‑тестов:
    - `model` — модель ИИ (пример: `Gemini 3 Pro`);
    - `mode` — режим (пример: `Разработка`, `Консультация`, `Чат`);
    - `userPrompt` — пользовательский запрос (ПАРАМЕТР 3);
    - `expectedText` — ожидаемый текстовый результат (ПАРАМЕТР 4);
  - для API‑тестов:
    - `method`, `url`, `requestBody`, `expectedStatus`, `expectedJsonPathChecks` и т.п.;
  - для UI‑тестов:
    - `page`, `userActions`, `expectedElements` и т.п.
- `expectedResult` (json) — формальное описание ожиданий (набор проверок/ассертов).
- `tags` (string[]) — произвольные теги (регрессия, критичный путь, багфикс‑NNNN и т.п.).
- `enabled` (boolean) — включён ли тест‑кейс.
- `createdAt`, `updatedAt`.

Дополнительно, для удобства работы тестировщика:

- поля «страницы/модули/функции системы» (структурированная детализация объекта тестирования):
  - `uiPages` — массив идентификаторов или путей страниц (например, `"/admin"`, `"/admin/tests"`), если сценарий касается UI;
  - `apiEndpoints` — список API‑маршрутов (например, `"/api/admin/agent/run"`, `"/api/items/[id]"`);
  - `codeRefs` — ссылки на ключевые файлы и функции (например, `src/app/admin/page.tsx`, `agent/core.ts`, имя функции);
  - `dbEntities` — список затрагиваемых моделей Prisma (`items`, `profiles`, `backlog` и т.д.).
- флаг и метаданные обогащения спецификации с ИИ:
  - `specEnrichedByAI` (boolean);
  - `specEnrichedAt` (datetime, nullable);
  - `specEnrichedModel` (string, например `"gemini-3.1-pro"`).

### 2.3. Прогоны тестов (TestRun)

Каждый запуск тест‑кейса фиксируется отдельной записью:

- `id` (uuid).
- `testCaseId` (fk → TestCase.id).
- `runNumber` (int) — порядковый номер для данного кейса.
- `startedAt`, `finishedAt`.
- `status` (enum):
  - `pending` — создан, но ещё не стартовал;
  - `running` — выполняется;
  - `success` — все проверки прошли;
  - `failed` — тест выполнен, но не прошёл ожидания;
  - `error` — техническая ошибка запуска (ошибка сети, исключение, некорректный ответ).
- `runner` (string) — источник запуска:
  - `admin-ui`;
  - `chat-command`;
  - `cron`;
  - `api`.
- `requestContext` (json) — фактические параметры запуска:
  - подставленные значения плейсхолдеров;
  - выбранная модель ИИ/режим;
  - HTTP‑запросы, если это API‑тест.
- `agentLogId` (string | null) — ссылка на лог рассуждений агента (если применимо).
- `statusDumpPath` (string | null) — путь к файлу дампа состояния.
- `conversationLog` (json, nullable) — история диалога с моделью ИИ (массив сообщений `{ role, content, timestamp }`).
- `steps` (json) — сводный «ход выполнения» теста (не путать с ходом выполнения агента):
  - массив шагов `{ type, title, detail, success }`.
- `comparisonResult` (json) — фактический результат сравнения с ожидаемым:
  - `success` (boolean);
  - `checks`: массив `{ name, ok, details }`.
- `diagnostics` (json) — произвольные диагностические данные (stacktrace, код ответа, содержимое важных полей).

## 3. Типы тестов и сценариев

Система должна поддерживать несколько типов сценариев, управляемых через единый каталог:

1. **Тесты ИИ‑агента и админки**  
   - Примеры: актуальность «хода выполнения», корректность pending/approval логики, интеграция с `run_command`.
   - Используют `promptTemplate`, параметры `model`, `mode`, `userPrompt`, `expectedText`.
   - Сценарий часто включает: «имитировать действия администратора в админке, выполнить запрос к `/api/admin/agent/run`, сравнить результат и шаги со списком ожиданий».

2. **API‑тесты**  
   - Напрямую проверяют `src/app/api/*` с помощью HTTP‑запросов.
   - В `parameters` фиксируются метод, URL, тело и ожидаемые статусы/фрагменты JSON.

3. **UI‑тесты** (минимальный уровень)  
   - Проверяют наличие/отсутствие элементов, корректность переходов по страницам.
   - В базовой реализации: выполняются полу‑автоматически через ИИ‑агента, который описывает шаги и проверяет ответы HTML/JSON через API (без полноценного браузер‑драйвера).

4. **Тесты Telegram‑бота**  
   - Имитация входящих обновлений Telegram (mock endpoint или отдельный тестовый бот).
   - Проверка формирования сообщений и логики работы с агентом.

5. **Тесты БД и фоновых задач**  
   - Проверка миграций, фоновых скриптов, целостности данных.
   - Используют Prisma и регламентированные internal‑API.

Каталог тест‑кейсов хранит **структуру и параметры**, а конкретная логика запуска инкапсулирована в исполнительном слое (см. раздел 6).

## 4. Каталог тест‑кейсов (структура хранения)

Хранение тест‑кейсов реализуется через:

- **БД (Prisma)** — основные сущности `TestModule`, `TestCase`, `TestRun`;
- **Документацию** — текущий файл с описанием архитектуры и возможными шаблонами сценариев;
- при необходимости — **seed‑скрипты** для начального набора ключевых тестов (регрессия критичных путей).

Минимальные требования:

- Все тест‑кейсы идентифицируются уникальным числом `number` (№ тест кейса).
- Для каждого теста можно:
  - посмотреть параметры (ПАРАМЕТР 1–4 и любые дополнительные);
  - увидеть список прогонов с результатами;
  - запустить новый прогон.

## 5. История прогонов и артефакты

Для каждого прогона фиксируются:

- статус и длительность;
- шаги выполнения;
- фактические входные данные (подставленные параметры);
- артефакты:
  - reasoning log агента (`agentLogId`);
  - статус‑дамп (`statusDumpPath`);
  - текстовые логи и JSON‑ответы.

Ссылки на артефакты:

- reasoning log: `/api/admin/agent/log?logId=...`;
- дампы: путь к файлу на сервере (`/home/ubuntu/komissionka/status-YYYYMMDD-HHMMSS.dump`) с возможностью скачивания из админки;
- детальная карточка прогона: `/admin/test-cases/run/[runId]`.

## 6. API и механика запуска

### 6.1. Admin API для каталога

Подсистема API будет расположена в `src/app/api/admin/test-cases/*` и обеспечивать:

- `GET /api/admin/test-cases` — список тест‑кейсов с краткой информацией и сводкой по последнему прогону.
- `POST /api/admin/test-cases` — создание нового тест‑кейса.
- `GET /api/admin/test-cases/[id]` — подробности тест‑кейса.
- `PUT /api/admin/test-cases/[id]` — изменение параметров.
- `DELETE /api/admin/test-cases/[id]` — отключение/удаление (мягкое).

### 6.2. Запуск тестов

Единый endpoint запуска:

- `POST /api/admin/test-cases/[id]/run`
  - Тело (минимально): `{ source?: "admin-ui" | "chat-command" }`.
  - Шаги:
    1. Найти тест‑кейс по `id`.
    2. Создать запись `TestRun` со статусом `pending`.
    3. Перевести статус в `running`, зафиксировать `startedAt`.
    4. Вызвать исполнитель (`runSingleTestCase(testCase, run)`), который:
       - подставляет параметры в шаблон промпта или HTTP‑запрос;
       - выполняет соответствующий сценарий (ИИ‑агент, HTTP к API, внутренние проверки);
       - собирает `steps`, `conversationLog`, `comparisonResult`, `diagnostics`, `agentLogId`, `statusDumpPath`.
    5. Обновить `TestRun`:
       - `status` → `success` / `failed` / `error`;
       - `finishedAt`.
    6. Вернуть структуру:
       ```json
       {
         "runId": "<uuid>",
         "status": "success",
         "summary": "Описание ключевых проверок",
         "checks": [ { "name": "containsExpectedText", "ok": true } ],
         "links": {
           "run": "/admin/test-cases/run/<uuid>",
           "agentLog": "/api/admin/agent/log?logId=...",
           "statusDump": "/download/status-20260319-110924.dump"
         }
       }
       ```

Дополнительно:

- `GET /api/admin/test-cases/[id]/runs` — список прогонов по кейсу.
- `GET /api/admin/test-cases/runs/[runId]` — подробная карточка прогона.

## 7. Интеграция с UI админки

В админке добавляется новая вкладка, например **«Тесты»**.

### 7.1. Список тест‑кейсов

Функциональность:

- Табличный список всех тест‑кейсов:
  - № теста;
  - модуль (с цветным бейджем);
  - название;
  - тип (`manual-guided` / `semi-automatic` / `automatic`);
  - краткое описание;
  - ПАРАМЕТР 1–4 (выводятся как отдельные колонки для ИИ‑тестов, для других типов — свои параметры);
  - **количество прогонов** (`runsCount`) — сколько раз этот тест‑кейс уже запускался;
  - **статус последнего прогона** (`lastStatus`) и его дата/время;
  - **% успешности** за всю историю (`successRate`) — отношение числа успешных прогонов к общему количеству запусков (отображать в виде «80% (8/10)»);
  - кнопки:
    - «▶ Запустить»;
    - «История»;
    - «Редактировать».

### 7.2. Карточка тест‑кейса

Отдельный экран/модальное окно:

- подробное описание;
- параметры (включая ПАРАМЕТР 1–4 для сценариев с ИИ);
- блок «Страницы и модули системы», показывающий:
  - связанные страницы UI;
  - API‑эндпоинты;
  - ссылки на файлы/функции;
  - модели данных;
- агрегированная статистика по прогонам:
  - общее количество запусков;
  - количество успешных и неуспешных;
  - % успешности;
- список последних прогонов (таблица `TestRun` с цветным статусом и ссылкой «Подробнее»);
- кнопка «▶ Запустить сейчас».

На карточке тест‑кейса доступна кнопка **«Обогатить спецификацию тест‑кейса с ИИ»**:

- при нажатии:
  - формируется запрос к ИИ‑агенту (модель **Gemini 3 Pro**) с текущим описанием и параметрами тест‑кейса;
  - агент, выступая в роли ведущего тестировщика, анализирует описание/промпт, определяет:
    - какие страницы UI задействованы;
    - какие API‑эндпоинты вызываются;
    - какие модули/файлы/функции и модели Prisma участвуют;
  - агент возвращает структурированный JSON с полями:
    - обновлённое, профессионально переписанное `description`;
    - заполненные/уточнённые `uiPages`, `apiEndpoints`, `codeRefs`, `dbEntities`;
    - комментарии/подсказки по покрытию;
- UI, получив ответ, показывает пользователю превью изменений и, после подтверждения, обновляет поля тест‑кейса и помечает его как `specEnrichedByAI = true`.

### 7.3. Карточка прогона

Отдельный экран:

- базовая информация (тест, модуль, статус, длительность, runner);
- список проверок (`comparisonResult.checks`) с цветной индикацией;
- сводка шагов (`steps`);
- ссылка на reasoning log, если есть (`agentLogId`);
- ссылка на дамп состояния, если есть (`statusDumpPath`);
- возможность развернуть `conversationLog` (история диалога с ИИ).

## 8. Интеграция с ИИ‑агентом (запуск по командам)

Системный промпт и инструменты агента дополняются правилами:

- Если администратор в чате пишет команду вида:
  - «ВЫПОЛНИ СЦЕНАРИЙ ТЕСТИРОВАНИЯ №N»;
  - «Запусти тест‑кейс №N»;
  - «Прогон теста №N».
- Агент:
  1. Парсит номер `N`.
  2. Вызывает специальный tool, который делает HTTP‑запрос:
     - `POST /api/admin/test-cases/by-number/[N]/run` (обёртка над основным endpoint запуска).
  3. Дожидается завершения прогона.
  4. Возвращает пользователю структурированный отчёт:
     - номер теста;
     - модуль;
     - статус;
     - ключевые проверки;
     - ссылки на карточку прогона, reasoning log, дамп состояния.

Агент НЕ реализует логику теста внутри себя напрямую — он выступает как **оркестратор**, который:

- инициирует запуск через admin‑API;
- читает результат;
- поясняет его человеку.

## 9. Этапы реализации

### Этап 1. Схема БД и Prisma

1. Добавить в `prisma/schema.prisma` модели:
   - `TestModule`;
   - `TestCase`;
   - `TestRun`.
2. Прогнать `npx prisma migrate dev` (локально/на сервере по регламенту).
3. Обновить документацию по модели данных в `docs/manual/*.md` (ссылки на новые сущности).

### Этап 2. Базовый Admin API

1. Реализовать CRUD для `TestCase`:
   - `GET/POST/PUT/DELETE /api/admin/test-cases`.
2. Реализовать чтение истории прогонов:
   - `GET /api/admin/test-cases/[id]/runs`;
   - `GET /api/admin/test-cases/runs/[runId]`.

### Этап 3. Механизм запуска теста

1. Реализовать функцию `runSingleTestCase(testCase, run)` на сервере:
   - поддержать минимум два вида тестов:
     - ИИ‑тесты (параметры: модель, режим, промпт, ожидаемый текст);
     - API‑тесты (метод, url, status, json‑проверки).
2. Реализовать endpoint:
   - `POST /api/admin/test-cases/[id]/run` с логикой создания и обновления `TestRun`.

### Этап 4. UI админки (вкладка «Тесты»)

1. Добавить вкладку в `src/app/admin/page.tsx`.
2. Реализовать:
   - список тест‑кейсов (таблица);
   - карточку теста с историей прогонов;
   - карточку прогона.
3. Добавить кнопки «▶ Запустить» с использованием `POST /api/admin/test-cases/[id]/run`.

### Этап 5. Интеграция с ИИ‑агентом

1. Добавить в инструменты агента новый tool (например, `run_test_case_number`), который:
   - принимает `testCaseNumber: number`;
   - вызывает admin‑API запуска и возвращает его JSON.
2. Обновить системный промпт:
   - описать формат команд пользователя («ВЫПОЛНИ СЦЕНАРИЙ ТЕСТИРОВАНИЯ №N»);
   - явно указать, что для таких команд нужно использовать новый инструмент.
3. В отчёте агента всегда возвращать:
   - статус прогона;
   - ключевые проверки;
   - ссылки.

### Этап 6. Базовый набор тест‑кейсов

1. Создать seed‑набор критичных тестов:
   - баг с актуальностью «хода выполнения» в админке (ИИ‑тест);
   - несколько API‑тестов (создание/удаление товара, новости, отзывы);
   - один‑два теста Telegram‑бота (обмен сообщениями, простейший сценарий).
2. Описать их в документации и при необходимости добавить seed‑скрипт.

### Этап 7. Расширение и автоматизация

1. Добавить возможность регулярного запуска набора тестов (регрессия) по cron через отдельный воркер или PM2‑процесс.
2. Добавить агрегированные отчёты:
   - панель с последними результатами по модулям;
