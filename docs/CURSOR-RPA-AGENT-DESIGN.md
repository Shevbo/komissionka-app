# RPA-агент для работы с чатом Cursor

## Цель

Автоматизировать взаимодействие с ИИ Cursor из внешней системы (например, админка «Комиссионка AI» или оркестратор):

1. **Ввод промпта** — передать текст промпта в окно чата Cursor (как copy-paste).
2. **Парсинг результата** — получить полный ответ агента Cursor (весь вывод после выполнения промпта).

Так можно использовать Cursor как «бэкенд» для запросов без 503 и с тем же качеством, что в редакторе.

---

## Ограничения среды

- **Cursor** — десктопное приложение (Electron). Чат агента — часть UI, а не отдельный HTTP API.
- Публичного API «отправить промпт → получить ответ» у Cursor нет.
- Поэтому единственный реалистичный путь без поддержки со стороны Cursor — **автоматизация UI (RPA)** или обход через **расширение/интеграцию**, если появятся точки входа.

## Целевая архитектура (RPA)

```
┌─────────────────┐     промпт       ┌──────────────────────┐     UI Automation      ┌─────────────┐
│  Админка /      │ ───────────────► │  RPA-агент           │ ─────────────────────► │   Cursor    │
│  оркестратор    │                  │  (отдельный процесс  │   вставка в чат,       │   (окно     │
│                 │ ◄─────────────── │   или сервис)        │   отправка, чтение     │    чата)    │
└─────────────────┘   полный ответ   └──────────────────────┘   ответа               └─────────────┘
```

- **Вход RPA-агента:** промпт (строка или файл), опционально таймаут.
- **Выход:** полный текст ответа агента Cursor (один блок или конкатенация сообщений).
- **Среда:** Windows (первая цель), при необходимости — описание расширения на macOS/Linux.

## Технический подход (UI Automation)

1. **Обнаружение окна Cursor**
  По заголовку/классу окна (например, «Cursor» или класс окна Electron). Windows: Win32 API / `pywinauto` / UI Automation (UIA).
2. **Поиск поля ввода чата**
  Дерево доступности (Accessibility Tree) или контроль по имени/роли (например, «edit» для поля ввода). Резерв: координаты/привязка к стабильным элементам (если Cursor даёт стабильные automation id).
3. **Ввод промпта**
  Установить фокус на поле ввода, очистить, вставить текст (буфер обмена или SendKeys). Предпочтительно буфер обмена — меньше проблем с раскладкой.
4. **Отправка**
  Нажать кнопку отправки или горячую клавишу (например, Enter / Ctrl+Enter — по документации или эмпирически).
5. **Ожидание ответа**
  Критерии: появление нового блока текста в области ответа, или таймаут. Варианты: опрос содержимого элемента «ответ», появление текста «Thinking» и его исчезновение и т.п.
6. **Парсинг результата**
  Найти в дереве доступности контейнер с ответом агента, извлечь текст (get value / get text). Если ответ разбит на несколько сообщений — конкатенация. Сохранить в файл или вернуть в вызывающий процесс (stdout / сокет / HTTP).
7. **Интерфейс для админки**
  RPA-агент может быть:  
  - **CLI:** `cursor-rpa --prompt "..." [--timeout 120]` → вывод в stdout;  
  - **HTTP-сервис:** `POST /run { "prompt": "..." }` → `{ "result": "..." }`;  
  - **Файловый обмен:** запись промпта в `prompt.txt`, чтение ответа из `response.txt` после готовности.

## Дорожная карта

### Фаза 0: Исследование (1–2 дня)

- Зафиксировать версию Cursor и ОС (Windows 10/11).
- Открыть чат агента в Cursor, включить инспектор доступности (Accessibility Insights / inspect.exe) и снять:
  - класс/имя окна приложения;
  - путь в дереве до поля ввода чата (роль, имя, automation id, если есть);
  - путь до области ответа (где отображается ответ агента);
  - горячая клавиша отправки сообщения.
- Проверить, есть ли у Cursor CLI или настройки для headless/automation (changelog, docs).
- Выбрать стек: Python (pywinauto / pyautogui / UIA) или Node (nut.js / node-ui-automation) в зависимости от экосистемы проекта.

**Результат:** отчёт (в этом же репозитории) с путями в дереве доступности, скриншотами/схемой и решением по стеку.

---

### Фаза 1: MVP — ввод промпта (3–5 дней)

- Репозиторий/папка: например `cursor-rpa/` в корне или отдельный репо.
- Минимальный скрипт (Python или Node):
  - активация окна Cursor по заголовку/классу;
  - поиск поля ввода чата (по дереву доступности или fallback на координаты);
  - вставка переданной строки (аргумент CLI или файл) через буфер обмена;
  - отправка (клавиша/кнопка).
- Конфиг: заголовок окна, таймаут ожидания окна, опционально путь к полю ввода (если фиксированный).
- Документация: как запускать, откуда брать промпт, требования (Cursor запущен, чат открыт).

**Результат:** стабильно работающий «ввод промпта в Cursor» из CLI/файла.

---

### Фаза 2: Парсинг ответа (3–5 дней)

- В дереве доступности найти контейнер с текстом ответа агента (одно сообщение или список).
- Реализовать ожидание «ответ готов»: появление/обновление текста, исчезновение индикатора загрузки или таймаут.
- Извлечь полный текст ответа (одним блоком или склейка нескольких сообщений).
- Вывод: stdout или запись в файл (например `cursor-response.txt`).
- Обработка ошибок: таймаут, окно не найдено, поле ввода не найдено — коды выхода и сообщения.

**Результат:** скрипт «промпт на вход → полный ответ на выход».

---

### Фаза 3: Интеграция с админкой (2–4 дня)

- Режим работы RPA-агента:
  - **Вариант A:** CLI, админка вызывает `child_process.exec`/`spawn` и читает stdout.
  - **Вариант B:** RPA как HTTP-сервис (например Flask/Fastify), админка дергает `POST /run` и получает JSON с полем `result`.
- В админке (или в конфиге агента «Комиссионка AI») добавить опцию «Бэкенд: Gemini API | Cursor RPA». При выборе Cursor — вызывать RPA (CLI или HTTP) вместо fetch на агента/Gemini.
- Таймауты и повторные попытки при недоступности RPA (Cursor закрыт и т.д.).

**Результат:** возможность выбирать Cursor как источник ответов в админке.

---

### Фаза 4: Устойчивость и мониторинг (2–3 дня)

- Логирование: старт/конец, таймауты, ошибки UI (элемент не найден).
- Повторные попытки при сбое (окно не в фокусе, Cursor обновился).
- Чеклист совместимости: протестировать на 2–3 минорных обновлениях Cursor и зафиксировать в доке.
- Краткий RUNBOOK: что делать, если RPA перестал находить окно/поле (обновить селекторы, перезапустить Cursor).

**Результат:** стабильный контур и инструкции по поддержке.

## Риски и митигация


| Риск                         | Митигация                                                                                            |
| ---------------------------- | ---------------------------------------------------------------------------------------------------- |
| Cursor меняет UI/доступность | Версионировать селекторы, хранить в конфиге; фаза 0 повторить при крупном апдейте.                   |
| Разное разрешение/масштаб    | Опираться на accessibility, а не на координаты; при необходимости — конфигурируемые offset’ы.        |
| Cursor не в фокусе / свёрнут | Явно активировать окно перед вводом; при таймауте — понятная ошибка «Cursor не отвечает».            |
| Длинный ответ обрезается     | Убедиться, что читаем весь контейнер (скролл при необходимости или чтение полного текста из дерева). |

## Критерии готовности

- **Фаза 1:** из CLI/файла передаётся промпт → он появляется в чате Cursor и отправляется.
- **Фаза 2:** после отправки скрипт дожидается ответа и возвращает полный текст в stdout/файл.
- **Фаза 3:** в админке можно выбрать «Cursor RPA» и получить ответ агента через этот контур.
- **Фаза 4:** есть логи и инструкции; при смене версии Cursor есть чеклист обновления селекторов.

После выполнения фаз 0–2 контур «импорт промпта в Cursor → парсинг всего результата» считается реализованным; фазы 3–4 делают его частью продукта и поддерживаемым в проде.
