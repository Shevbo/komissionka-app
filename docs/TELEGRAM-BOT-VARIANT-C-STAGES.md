# Вариант C: Локальный агент в репозитории «Комиссионка» — этапы реализации

Задача: создать AI-помощника, который «живёт» в папке с кодом проекта, получает промпт, выполняет его в контексте репозитория (чтение файлов, поиск, при необходимости — запуск команд) и возвращает текстовый результат. Позже этого агента вызывает Telegram-бот.

Ниже — разбиение на **логичные этапы** с входными/выходными результатами и зависимостями.

---

## Общая схема этапов

```
Этап 1          Этап 2           Этап 3           Этап 4           Этап 5           Этап 6
Контракт   →   Инструменты  →   Ядро агента  →   Запуск/жизненный →   API для бота  →   Безопасность
и среда         (tools)          (LLM + цикл)     цикл
```

---

## Этап 1. Контракт агента и окружение в репозитории

**Цель:** определить, где агент живёт в проекте, как его вызывать и в каком формате обмениваться данными.

### 1.1 Размещение в репозитории

- Выделить каталог под агента, например: `komissionka/agent/` (или `packages/agent/`, если репозиторий станет монорепо).
- Внутри: свой `package.json`, зависимости (TypeScript, рантайм для вызова LLM, при необходимости MCP SDK), точка входа (например, `agent/index.ts` или `agent/run.ts`).
- Корень проекта считать «рабочей директорией» агента: все пути к файлам — относительно корня репозитория.

**Результат этапа:** структура папок и скрипт запуска агента из корня проекта (например, `npm run agent:start` или `node agent/run.js`).

### 1.2 Контракт «вход — выход»

- **Вход:** один текстовый промпт (строка). Опционально: параметры (макс. длина ответа, таймаут).
- **Выход:** один текстовый ответ (финальная выдача для пользователя). Опционально: структурированный лог шагов (только для отладки/логов).
- Решить формат вызова на этом этапе:
  - **Вариант 1:** агент как HTTP-сервис: принимает POST с телом `{ "prompt": "..." }`, возвращает `{ "result": "..." }`.
  - **Вариант 2:** агент как процесс: stdin — промпт (или путь к файлу с промптом), stdout — финальный ответ; бот или другой сервис запускает процесс и читает stdout.
  - **Вариант 3:** агент как MCP-сервер: клиент (бот или отдельный воркер) подключается к MCP и вызывает инструменты + LLM через протокол MCP.

**Результат этапа:** документ или комментарии в коде с описанием контракта (вход, выход, способ вызова). Реализация «заглушки»: на любой промпт возвращать фиксированную строку (например, «Agent stub»), чтобы следующий этап уже опирался на реальный интерфейс.

### 1.3 Окружение и конфигурация

- Переменные окружения: путь к корню проекта, при использовании облачной LLM — API-ключ, модель, таймауты.
- Файл конфигурации (опционально): `agent/config.json` или переменные в `.env` с префиксом `AGENT_`.

**Результат этапа:** список переменных окружения и способ их передачи при запуске агента (локально и на сервере).

## Этап 2. Инструменты агента (tools)

**Цель:** дать агенту возможность «видеть» проект: читать файлы, искать по коду, при необходимости — вызывать безопасные команды.

### 2.1 Базовые инструменты чтения

- **read_file(path)** — прочитать содержимое файла по пути относительно корня репозитория. Ограничить допустимые расширения или каталоги (например, только `src/`, `prisma/`, конфиги), запретить чтение за пределами репозитория.
- **list_dir(path)** — список файлов и папок в каталоге (без рекурсии или с ограниченной глубиной).
- **glob(pattern)** или **find_files(pattern)** — поиск файлов по маске (например, `**/*.ts`, `prisma/*.prisma`).

**Результат этапа:** реализованные функции/инструменты и их описание для LLM (имена, параметры, что возвращают). Тесты на реальных путях внутри репозитория.

### 2.2 Поиск по содержимому

- **grep(search_string, path?, options?)** — поиск по тексту в файлах (регулярное выражение или подстрока), ограничение по каталогам и типам файлов. Возвращать совпадения с контекстом (несколько строк вокруг).

**Результат этапа:** инструмент поиска, интегрированный в набор tools агента.

### 2.3 Опционально: безопасное выполнение команд

- **run_command(command, cwd?, timeout?)** — выполнить разрешённую команду в корне проекта (или в указанной подпапке). Белый список команд: например, `npx prisma generate`, `npm run build`, `npx tsc --noEmit`, чтение вывода (stdout/stderr). Запрет на интерактивные команды и команды с произвольным вводом.
- Желательно запуск в изолированной среде (таймаут, при возможности — ограничение по памяти) и без доступа к секретам из `.env` в окружении процесса.

**Результат этапа:** либо реализованный безопасный `run_command` с белым списком, либо явное решение «без выполнения команд» с документированием ограничений.

### 2.4 Описание инструментов для LLM

- Сформировать для каждой функции описание в формате, ожидаемом выбранным LLM-провайдером (OpenAI function calling, Anthropic tools, MCP tools и т.д.): имя, описание, параметры (тип, обязательность).

**Результат этапа:** готовый набор tools и их JSON-схема/описание для использования на этапе 3.

## Этап 3. Ядро агента (цикл «промпт → контекст → LLM → ответ»)

**Цель:** реализовать логику «получил промпт → при необходимости вызвал инструменты → получил ответ от LLM → вернул пользователю финальный текст».

### 3.1 Выбор LLM и способа вызова

- Определить источник LLM: облачный API (OpenAI, Anthropic, и т.п.) или локальная модель (Ollama, LM Studio). Учесть ограничения по длине контекста и стоимости.
- Реализовать тонкую обёртку вызова: системный промпт + пользовательский промпт + (при использовании tool use) результаты вызовов инструментов.

**Результат этапа:** один модуль «клиент LLM» с интерфейсом: `request(messages, tools?) => response`.

### 3.2 Системный промпт

- Сформировать системный промпт, в котором зафиксировано: агент — помощник по проекту «Комиссионка», стек (Next.js, Prisma, PostgreSQL), что можно делать (читать файлы, искать, при наличии — запускать разрешённые команды) и что возвращать (краткий, понятный текст для пользователя).

**Результат этапа:** стабильная строка/шаблон системного промпта в коде или конфиге.

### 3.3 Цикл рассуждения и вызова инструментов

- Реализовать цикл: отправить в LLM промпт пользователя (и при необходимости историю вызовов tools); если LLM запрашивает вызов инструмента — выполнить его в репозитории и добавить результат в диалог; повторять до финального ответа или лимита шагов.
- Ограничить число итераций (например, не более 5–10 вызовов tools на один запрос), чтобы избежать зацикливания и лишних затрат.

**Результат этапа:** функция вида `runAgent(prompt: string): Promise<string>`, возвращающая только финальный текст ответа пользователю.

### 3.4 Обработка ошибок и таймаутов

- Таймаут на весь запрос к агенту (например, 60–120 секунд).
- При ошибке чтения файла, вызова LLM или инструмента — возвращать короткое сообщение об ошибке пользователю (без внутренних деталей в ответе), логировать детали в сервисные логи.

**Результат этапа:** стабильное завершение с таймаутом и понятными сообщениями об ошибках.

## Этап 4. Запуск и жизненный цикл процесса агента

**Цель:** определить, как агент «живёт» на машине: постоянно работающий процесс или запуск по запросу.

### 4.1 Режим работы

- **Вариант A — долгоживущий процесс:** агент поднимается как HTTP-сервер (или MCP-сервер) при старте и обрабатывает запросы по мере поступления. Удобно для интеграции с ботом (один раз поднял — много запросов).
- **Вариант B — процесс на запрос:** при каждом запросе от бота запускается отдельный процесс агента (stdin/stdout или разовый HTTP), по завершении — процесс завершается. Проще изоляция, выше задержка на старт.

**Результат этапа:** выбран режим и описан в конфигурации/документации.

### 4.2 Запуск в dev и на сервере

- Локально: `npm run agent:start` (или аналог) из корня проекта; при HTTP — проверка через `curl` с тестовым промптом.
- На сервере: запуск через PM2, systemd или Docker с теми же переменными окружения и рабочим каталогом = корень репозитория.

**Результат этапа:** инструкция по запуску в `docs/` или в README агента; при необходимости — пример unit/systemd или Docker.

## Этап 5. API для вызова из Telegram-бота

**Цель:** бот может вызвать агента по сети (локально или на том же сервере) и получить финальную выдачу.

### 5.1 Эндпоинт вызова агента

- Если агент — HTTP-сервис: endpoint, например `POST /run` с телом `{ "prompt": "..." }`, ответ `{ "result": "..." }` или `{ "error": "..." }`.
- Аутентификация: внутренний API-ключ или проверка по заголовку/секрету, чтобы вызывать мог только сервис бота (или доверенные сервисы). Не выставлять этот endpoint в публичный интернет без защиты.

**Результат этапа:** стабильный контракт API и способ аутентификации, описанные в документации.

### 5.2 Интеграция с ботом

- В коде Telegram-бота заменить вызов внешнего AI API (Вариант B) на вызов локального агента: HTTP-запрос к `http://localhost:PORT/run` (или к внутреннему URL сервера) с промптом из сообщения.
- Обработать таймаут и ошибки: при недоступности агента — сообщение пользователю «Сервис временно недоступен» и при необходимости fallback на Вариант B (облачный API).

**Результат этапа:** бот использует Вариант C при доступности агента и корректно возвращает ответ в Telegram (с разбивкой по длине сообщения, как в общем дизайне бота).

## Этап 6. Безопасность, лимиты и наблюдаемость

**Цель:** минимизировать риски и предсказуемо ограничить использование ресурсов.

### 6.1 Ограничения на инструменты

- Жёсткий запрет чтения файлов вне репозитория (проверка путей, отсутствие `..` и симлинков за пределы корня).
- Для `run_command` — только белый список команд, без подстановки пользовательского ввода в команду (или строгая санитизация).
- При необходимости — read-only режим для части инструментов (например, только чтение, без записи в файлы).

**Результат этапа:** проверки в коде инструментов и краткий раздел «Безопасность» в документации агента.

### 6.2 Лимиты и таймауты

- Лимит длины промпта (например, 4–8 КБ).
- Лимит длины ответа LLM (например, 4–8 КБ) или обрезка перед возвратом в бота.
- Таймаут на один запрос к агенту (см. этап 3); при превышении — завершение с сообщением «Превышено время ожидания».

**Результат этапа:** конфигурируемые лимиты и их применение в коде.

### 6.3 Логирование и мониторинг

- Логировать: факт запроса (без полного текста промпта при необходимости), длительность, успех/ошибка, количество вызовов инструментов. Не логировать в открытом виде секреты и полные ответы LLM, если они чувствительны.
- При запуске в production — возможность подключить метрики (число запросов, ошибок, задержка).

**Результат этапа:** описание формата логов и при необходимости пример интеграции с системой мониторинга.

## Рекомендуемый порядок выполнения

1. **Этап 1** — сделать первым: без контракта и структуры нельзя стабильно разрабатывать остальное.
2. **Этапы 2 и 3** можно частично распараллелить: сначала минимальный набор tools (read_file, list_dir), затем первый цикл с LLM; потом добавить grep и при необходимости run_command.
3. **Этап 4** — после появления рабочего `runAgent` на этапе 3.
4. **Этап 5** — после того как агент стабильно запускается и отвечает локально (этап 4).
5. **Этап 6** — внедрять по мере появления инструментов и перед выводом в production; критичные проверки безопасности — до этапа 5.

После завершения всех этапов Вариант C готов к использованию Telegram-ботом как основной или дополнительный способ выполнения промптов в контексте проекта «Комиссионка».
