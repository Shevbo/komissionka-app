# Запуск проекта в РФ: хостинг и хранение данных в России

Пошаговая инструкция по выбору услуг и переносу проекта (Next.js + Prisma + PostgreSQL) на российский хостинг с хранением данных в РФ. Использование **Prisma** упрощает смену БД: достаточно поменять `DATABASE_URL` и при необходимости провайдера драйвера.

---

## 1. Какие услуги искать у поставщиков (спецификация)

### 1.1 Хостинг приложения (Next.js / Node.js)

| Что искать | Спецификация (эффективно и экономно) |
|------------|--------------------------------------|
| **Услуга** | VPS (виртуальный сервер) или облачный сервер (Cloud Server / Compute) с данными в РФ |
| **ОС** | Linux (Ubuntu 22.04 LTS или 24.04 LTS) |
| **Минимум для старта** | 1 vCPU, 1–2 ГБ RAM, 20–30 ГБ SSD/NVMe |
| **Рекомендуемый минимум** | 2 vCPU, 2–4 ГБ RAM, 40–50 ГБ SSD — для Next.js + фоновых процессов |
| **Сеть** | Исходящий трафик: уточнять лимиты (часто 1–3 ТБ/мес входят в тариф) |
| **Соответствие** | 152-ФЗ (персональные данные в РФ), дата-центр в России (Москва, СПб и т.д.) |

**Технологии:** Node.js 20 LTS, PM2 или systemd для запуска `npm start`, Nginx (или аналог) как reverse proxy, SSL (Let's Encrypt или сертификат от хостера).

### 1.2 База данных (PostgreSQL)

| Вариант | Когда выбирать | Спецификация |
|---------|----------------|---------------|
| **Managed PostgreSQL (DBaaS)** | Предпочтительно: не нужно администрировать СУБД | 1 vCPU, 2 ГБ RAM, 20–40 ГБ SSD; бэкапы; данные в РФ |
| **PostgreSQL на своём VPS** | Максимальная экономия, готовы сами ставить и обновлять СУБД | Отдельный VPS или тот же сервер, что и приложение (для малой нагрузки) |

**Технологии:** PostgreSQL 15 или 16; для Prisma достаточно строки подключения (логин, пароль, хост, порт, имя БД). Миграции: `npx prisma migrate deploy`.

### 1.3 Итог по услугам для поиска

- **Хостинг приложения:** «VPS в России», «облачный сервер (Cloud Server) с дата-центром в РФ», «Node.js хостинг».
- **БД:** «Managed PostgreSQL в России», «PostgreSQL как сервис (DBaaS)», либо «VPS с установленным PostgreSQL».
- **Дополнительно:** домен (если нужен), SSL, опционально — объектное хранилище (S3-совместимое) для загрузок файлов (у вас пока `public/uploads`).

## 3. Рекомендации: каким сервисом и каким поставщиком пользоваться

### Общая схема

- **Приложение:** один VPS/облачный сервер в РФ (Node.js, Next.js, Nginx, SSL).  
- **БД:** Managed PostgreSQL в РФ — меньше возни с обновлениями и бэкапами.  
- **Альтернатива для минимального бюджета:** один VPS в РФ, на нём и приложение, и PostgreSQL (для небольшой нагрузки и малого числа пользователей).

### Рекомендуемый вариант для новичка и баланса «цена/простота»

**Timeweb Cloud**

- Понятная панель, русская документация и поддержка (чат, тикеты, Telegram).  
- Managed PostgreSQL с фиксированными тарифами (например, от 790 ₽/мес за минимальный инстанс) — не нужно считать почасовку.  
- Дата-центры в РФ, заявлено соответствие 152-ФЗ.  
- Отдельно заказать облачный сервер (VPS) для приложения — на том же портале timeweb.cloud, всё в одном аккаунте.

**Схема:**

1. **БД:** Managed PostgreSQL (тариф хотя бы 1 vCPU, 2 ГБ RAM, 20 ГБ) — получаете хост, порт, логин, пароль, имя БД.  
2. **Приложение:** облачный сервер (1–2 vCPU, 2 ГБ RAM, 20–40 ГБ) — развернуть Node.js, Next.js, настроить Nginx и SSL.

**Альтернативы:**

- **Selectel** — если важна максимальная «серьёзность» инфраструктуры и готовность считать по калькулятору; есть бонусы на тест.  
- **VK Cloud** — удобно, если уже пользуетесь экосистемой VK; цены по факту использования.  
- **Reg.ru** — если нужен дёшево только VPS в РФ и вы готовы сами установить PostgreSQL на тот же сервер (без Managed DB).

## 4. Пошаговая инструкция для новичка: переход на новый хостинг (по рекомендации — Timeweb Cloud)

Ниже — упрощённый порядок действий. Детали интерфейса уточняйте в актуальной документации Timeweb Cloud.

### Шаг 1. Регистрация и создание БД (Managed PostgreSQL)

1. Зарегистрируйтесь на https://timeweb.cloud.  
2. В разделе «Сервисы» → «Cloud Databases» (или «Managed PostgreSQL») создайте кластер PostgreSQL.  
3. Выберите тариф (для старта подойдёт минимальный, например 1 vCPU, 2 ГБ RAM, 20 ГБ).  
4. Укажите регион/дата-центр **Россия**.  
5. Задайте имя БД, логин и пароль пользователя БД (сохраните их в надёжном месте).  
6. После создания скопируйте **хост** (например, `xxx.pg.timeweb.cloud`), **порт** (обычно 5432), **имя БД**, **логин**, **пароль**.

Строка подключения будет такой (подставьте свои значения):

```text
DATABASE_URL="postgresql://ЛОГИН:ПАРОЛЬ@ХОСТ:ПОРТ/ИМЯ_БД?sslmode=require"
```

Часто в панели дают готовую строку или подсказку для `sslmode` — используйте её.

### Шаг 2. Создание сервера для приложения (VPS)

1. В том же личном кабинете Timeweb Cloud откройте раздел «Облачные серверы» (или «VPS»).  
2. Создайте сервер:  
   - ОС: **Ubuntu 22.04** или **24.04 LTS**.  
   - Регион: **Россия**.  
   - Конфигурация: минимум 1 vCPU, 2 ГБ RAM, 20–30 ГБ диска (для роста можно 2 vCPU, 4 ГБ).  
   - **Версия PHP:** если панель требует выбрать PHP — наш проект (Next.js) **не использует PHP**, работает на Node.js. Можно выбрать любую (например **PHP 8.3**) или пропустить, если есть опция «Без PHP» / «Node.js». PHP на сервере не помешает — Node.js ставится отдельно (Шаг 3).  
3. Сохраните логин и пароль (или SSH-ключ) для доступа по SSH.  
4. Привяжите к серверу **публичный IP** (часто включается в тариф или добавляется опцией).

### Шаг 3. Подключение к серверу и установка окружения

1. Подключитесь по SSH (с вашего ПК):

   ```bash
   ssh root@ВАШ_IP_СЕРВЕРА
   ```

   Если провайдер выдал другого пользователя (например, `srv12345` вместо root), подставьте его: `ssh srv12345@ВАШ_IP_СЕРВЕРА`.

2. Обновите пакеты и установите Git (нужен для `git clone` в Шаге 4):

   ```bash
   sudo apt update && sudo apt install -y git
   ```

3. Установите Node.js 20 LTS (поддерживает Ubuntu 22.04 и 24.04):

   ```bash
   curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
   sudo apt-get install -y nodejs
   ```

4. Установите Nginx (reverse proxy и SSL):

   ```bash
   sudo apt install -y nginx
   ```

5. (Опционально) Установите PM2 для запуска приложения:

   ```bash
   sudo npm install -g pm2
   ```

### Шаг 4. Загрузка проекта на сервер

1. На своём компьютере соберите проект и при необходимости упакуйте (или клонируйте с Git на сервер):

   ```bash
   npm run build
   ```

2. Перенесите на сервер папку проекта (через SCP, SFTP, Git или архив). Пример через SCP с локальной машины (из каталога проекта):

   ```bash
   scp -r . root@ВАШ_IP_СЕРВЕРА:/home/nextjs-app
   ```

   Либо на сервере: `git clone ВАШ_РЕПОЗИТОРИЙ /home/nextjs-app`, затем `cd /home/nextjs-app`, `npm install` и `npm run build`.

### Шаг 5. Переменные окружения и Prisma на сервере

1. На сервере в папке проекта создайте файл `.env` (или `.env.production`):

   ```bash
   cd /home/nextjs-app  # или ваш путь
   nano .env
   ```

2. Задайте переменные (подставьте свои значения):

   ```env
   DATABASE_URL="postgresql://ЛОГИН:ПАРОЛЬ@ХОСТ_ИЗ_ШАГА_1:5432/ИМЯ_БД?sslmode=require"
   NEXTAUTH_SECRET="длинный-случайный-секрет-не-менее-32-символов"
   NEXTAUTH_URL="https://ваш-домен.ru"
   ```

3. Установите зависимости и сгенерируйте Prisma Client, примените миграции:

   ```bash
   npm install
   npx prisma generate
   npx prisma migrate deploy
   ```

   Если в проекте используется seed и он нужен на проде (обычно не для прода), можно выполнить:  
   `npm run seed`.

### Шаг 6. Запуск приложения

1. Запуск через PM2 (чтобы приложение не падало при отключении SSH):

   ```bash
   pm2 start npm --name "komissionka" -- start
   pm2 save
   pm2 startup
   ```

   Либо без PM2 (только для проверки):  
   `npm start` — приложение будет слушать порт 3000.

### Шаг 7. Настройка Nginx и SSL

1. Создайте конфиг Nginx (например, `/etc/nginx/sites-available/komissionka`):

   ```nginx
   server {
       listen 80;
       server_name ваш-домен.ru www.ваш-домен.ru;
       location / {
           proxy_pass http://127.0.0.1:3000;
           proxy_http_version 1.1;
           proxy_set_header Upgrade $http_upgrade;
           proxy_set_header Connection 'upgrade';
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
           proxy_set_header X-Forwarded-Proto $scheme;
           proxy_cache_bypass $http_upgrade;
       }
   }
   ```

2. Включите сайт и перезапустите Nginx:

   ```bash
   sudo ln -s /etc/nginx/sites-available/komissionka /etc/nginx/sites-enabled/
   sudo nginx -t && sudo systemctl reload nginx
   ```

3. SSL (HTTPS): на Timeweb часто есть встроенный SSL или инструкция по Let's Encrypt (certbot). Установите сертификат и измените конфиг на `listen 443 ssl;` и при необходимости редирект с 80 на 443.

### Шаг 8. Домен и загрузки

1. В панели управления доменом у регистратора укажите A-запись на публичный IP вашего сервера.  
2. Загрузки файлов: сейчас проект сохраняет файлы в `public/uploads`. На сервере убедитесь, что папка существует и права записи у процесса Node (или настройте отдельное хранилище позже).

### Шаг 9. Проверка

1. Откройте в браузере `https://ваш-домен.ru`.  
2. Проверьте вход, каталог, корзину, админку.  
3. В Prisma Studio при необходимости можно проверить данные (на своей машине с `DATABASE_URL` продакшен-БД или через безопасный туннель).

## 5. PHP или Node.js?

**PHP** — язык для классического веб-хостинга (WordPress, Laravel и т.п.).  
**Node.js** — среда, на которой работает Next.js и этот проект.

Если хостинг при создании сервера предлагает выбрать версию PHP (5.6, 7.x, 8.x):

- Для **Комиссионки** PHP **не нужен**: приложение запускается через `npm start` на Node.js.
- Если выбор PHP обязателен (панель не даёт пропустить) — укажите **PHP 8.2** или **8.3** (последние стабильные). Это не помешает работе Node.js.
- Node.js ставится отдельно по SSH (см. Шаг 3). PHP можно игнорировать.

## 6. Важные замечания

- **Безопасность:** не коммитьте `.env` в Git; на сервере храните секреты только в переменных окружения или защищённых конфигах.  
- **Резервное копирование:** у Managed PostgreSQL бэкапы обычно включены; дополнительно можно настроить дампы (pg_dump) по расписанию.  
- **Миграции:** после любых изменений схемы Prisma на своей машине выполните `npx prisma migrate dev`, закоммитьте папку `prisma/migrations`, на сервере после деплоя — только `npx prisma migrate deploy`.  
- Цены и контакты в документе приведены по состоянию поиска; перед заключением договора проверяйте актуальные тарифы и реквизиты на сайтах поставщиков.

Использование **Prisma** действительно упрощает смену хостинга и БД: достаточно новой `DATABASE_URL` и при необходимости смены драйвера/адаптера в коде; структура запросов и миграций остаётся той же.
